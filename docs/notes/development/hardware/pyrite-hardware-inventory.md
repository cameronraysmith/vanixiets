# pyrite hardware inventory

## Scope and provenance

This note records what was observed on the machine designated `pyrite`, an Apple MacBookPro14,1, during live reconnaissance on 2026-07-16.
Everything below was read from the running machine over ssh while it was booted from a stock NixOS 26.05 installer ISO on kernel 7.1.3, with no configuration applied and macOS still resident on the internal disk.

Two different kernels are in play throughout this note, and statements about them are not in conflict.
The 7.1.3 above is the installer ISO's: `nixos/modules/installer/cd-dvd/latest-kernel.nix:3` sets `boot.kernelPackages = pkgs.linuxPackages_latest`.
An installed pyrite instead takes the `boot.kernelPackages` default, which resolves through `linuxPackages` to `pkgs/top-level/linux-kernels.nix:745` (`linux_default = linux_6_18`) and lands on 6.18.37 in this repository's pinned nixpkgs.
No module pyrite imports overrides that default — the repository's one `boot.kernelPackages` assignment, at `modules/nixos/k3s-server/kernel.nix:19`, is gated on `config.k3s-server.enable` and never reaches this machine — and the nixos-hardware profile's own `linuxPackages_latest` at `apple/macbook-pro/14-1/default.nix:26` is gated on `lib.versionOlder pkgs.linux.version "6.0"` and so is dead code at 6.18.37.

The distinction between this note and `mbp141-nixos-clan-research.md` is authorship of the evidence.
That report reasons from a third-party dump corpus and from kernel source; this note reports the unit itself.
Where the two disagree, this note is the authority.

The raw material is retained beside this note rather than inlined.
`pyrite-recon-raw.txt` is the 884-line command dump, and `pyrite-dmesg-raw.txt` is the 1025-line root dmesg.
A machine-readable hardware report generated by nixos-facter lives with the `pyrite-baremetal-nixos` OpenSpec change and moves to `machines/pyrite/facter.json` when that change is applied.

## Identity

The machine reports itself as `MacBookPro14,1` from Apple Inc., which is the 2017 13-inch non-Touch-Bar model.
It carries neither a T1 nor a T2 coprocessor.
It boots external media from the Option-key picker, and no firmware password is set — this was established by booting the installer ISO, which is the observation the entire single-boot plan was gated on.

The CPU is Intel Kaby Lake-U with GT3 graphics, presenting as Iris Plus Graphics 640 at PCI ID `8086:5926`.
The i915 driver loads DMC firmware `i915/kbl_dmc_ver1_04.bin` unaided.

## Storage, and the two footguns it carries

The controller is an Apple S3X NVMe Controller at PCI ID `106b:2003` revision 12, PCI class `0x0180`.
The disk is an APPLE SSD AP0512J, serial `C08843605KKHV4MAK`, 465.92 GiB (500277792768 bytes), GPT-partitioned.
At recon time it held a 300 MiB EFI system partition and a 465.6 GiB APFS partition carrying macOS, both of which the install destroys.

Two properties of this device diverge from every other machine in the fleet and each is a distinct hazard.

The disk reports a logical sector size of 4096 bytes, not 512, and its physical sector size is likewise 4096.
No machine in this repository sets `options.ashift` on any zpool, because every existing NixOS machine is a cloud VM on a 512-byte-sector virtual disk where ZFS's autodetection lands on the right answer regardless.
On a 4Kn device the pool must be created with `ashift = "12"` explicitly; disko performs no sector-size detection of its own and passes zpool options through verbatim.

The controller exposes two namespaces.
`nvme0n1` is the 465.9 GiB disk.
`nvme0n2` is 8 KiB and must never be written to.
Both are reachable under `/dev/disk/by-id/` with names sharing a common prefix:

```
nvme-APPLE_SSD_AP0512J_C08843605KKHV4MAK      -> nvme0n1
nvme-APPLE_SSD_AP0512J_C08843605KKHV4MAK_1    -> nvme0n1
nvme-APPLE_SSD_AP0512J_C08843605KKHV4MAK_2    -> nvme0n2   (the 8 KiB namespace)
nvme-nvme.106b-4330383834333630354b4b4856344d414b-4150504c4520535344204150303531324a-00000001 -> nvme0n1
```

The unsuffixed name and the `_1` name were both observed resolving to `nvme0n1`.
The `_1` form is namespace-explicit, which is why the disko layout uses it: the unsuffixed name is a bare prefix of both suffixed names, so any prefix or glob match over this family can reach `_2`.

The controller reports `1/0/0 default/read/poll queues`.

## Networking

WiFi is a Broadcom BCM4350 at PCI ID `14e4:43a3` revision 05, Apple subsystem `106b:0170`, driven by `brcmfmac`.
Firmware loaded unaided from the stock ISO: the kernel logged `using brcm/brcmfmac4350c2-pcie for chip BCM4350/5` and firmware version `7.35.180.133`, and the interface associated to a real access point with no intervention.
No firmware was extracted from macOS, and none is needed.
The blob ships in `linux-firmware` under a redistributable licence, so this hardware needs no unfree package.

One cosmetic gap accompanies it: `brcmf_c_process_clm_blob: no clm_blob available (err=-2), device may have limited channels available`.

Bluetooth presents as `hci0`, a BCM4350C0.
The kernel logs `Bluetooth: hci0: BCM: firmware Patch file not found, tried: 'brcm/BCM.hcd'`.
Whether Bluetooth is functional was not tested and remains open.

The only non-loopback interface is `wlp2s0`.
This machine has no wired NIC and no virtio devices.

## Input, and what it implies for the boot passphrase

The internal keyboard and trackpad are SPI-attached and driven by mainline `applespi`, which logged `applespi spi-APP000D:00: modeswitch done`.
The observed module set is `applespi`, `spi_pxa2xx_platform`, `spi_pxa2xx_core`, `intel_lpss_pci`, `intel_lpss`, and `applesmc`.

The consequence is that a passphrase prompt on the internal keyboard at stage-1 initrd is viable, provided the initrd module set includes those modules.
It is not free: the fleet's shared base module supplies only `virtio_pci` and `virtio_net` to initrd, so pyrite must carry the SPI modules itself or import a profile that does.

## Power, thermal, and peripherals

`applesmc` works; a fan and a temperature sensor were readable (fan 1, temp 33).
Both `BAT0` and `ADP1` are present.

The camera is a Broadcom 720p FaceTime HD at PCI ID `14e4:1570`.
Its driver, `facetimehd`, is unfree and out-of-tree, and the camera was not exercised.

## Audio

Audio is not a work item, by binding decision, recorded at `pyrite-baremetal-nixos/design.md:145` under Non-Goals.
It is recorded here as a hardware fact rather than as a defect to fix.

The codec is a Cirrus Logic CS8409, vendor ID `0x10138409`, subsystem ID `0x106b3300`, on an Intel Sunrise Point-LP HD Audio controller at `8086:9d71`.
On the ISO's 7.1.3 the mainline `snd_hda_codec_cs8409` driver binds and autoconfigures — line outs 2 of type speaker, one headphone out, internal mic at `0x44`, mic at `0x3c` — and the PCM reaches state RUNNING with advancing `hw_ptr` and `appl_ptr`, while producing no audible sound.
Silence was confirmed by a human listening to the machine.

The amplifier initialization sequence is not upstream, which is what the running-but-silent PCM reflects.
The out-of-tree module that does carry it has a tested ceiling of kernel 6.17 against the 6.18.37 an installed pyrite takes from this repository's pinned nixpkgs, and pinning the kernel backward is ruled out.
Audio is therefore a watch-item on upstream: if mainline ever lands the amplifier init, it arrives on a kernel bump at no cost.

## Corrections to prior inference

Three conclusions in `mbp141-nixos-clan-research.md` do not survive contact with the machine or with the repository as it stands.

The claim that the nixos-hardware `apple-macbook-pro-14-1` profile's `b43Firmware` pull hard-fails evaluation is wrong.
It fails only under `allowUnfree = false`, and `modules/nixpkgs/base-defaults.nix:23` sets `allowUnfree = true` for every machine in the fleet, so the profile evaluates.
The b43 pull remains a misdetection — b43 firmware is for entirely different silicon than the BCM4350 this machine carries — and is worth disabling as a correction, but it is not an evaluation blocker.

The audio conclusion is superseded by the binding decision above.

Every nixos-hardware claim in this note is provisional on a reading that is not pinned.
The input does not appear in `flake.nix` or `flake.lock`, so no locked revision exists to read it at; the readings here come from a local checkout at HEAD `fccfa90` (2026-07-15) and can drift from whatever revision an evaluation actually resolves.

nixos-facter versions 0.4.4 and 0.4.5-dev both fail on this machine with `unsupported bus type: Spi`, which is `applespi`.
PR nix-community/nixos-facter#672 (branch `fix/spi-touchpad-bus`, open and mergeable as of 2026-07-15, fixing issue #339 which was filed against this exact model) does work: it produced a valid 80 KB report on this machine.
Because a facter report is a static artifact that clan-core consumes by path existence, the unmerged PR does not block installing this machine — it blocks only regenerating the report, including via `clan machines update-hardware-config`.

Separately, `nixos-generate-config` on this machine imports `<nixpkgs/nixos/modules/hardware/network/broadcom-43xx.nix>`.
That is the same b43 misdetection arriving by a second route, and it is the reason the machine's configuration must not be seeded from `nixos-generate-config` output.

## Open questions

Whether Bluetooth functions is untested; the missing `brcm/BCM.hcd` patch file is logged but its consequence was not observed.

Whether this unit needs the `d3cold` suspend workaround documented for Apple-NVMe MacBookPro13,x and 14,x models, and whether it helps, was not tested — the machine was never suspended during recon.
