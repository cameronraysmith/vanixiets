# buildbot-nix configuration

This reference documents the per-repo and server-side configuration for buildbot-nix, covering the `buildbot-nix.toml` schema, GitHub App setup, worker configuration, the dual-scheduler architecture, the effect pipeline, and merge gating.


## `buildbot-nix.toml` schema

The per-repo configuration file lives at the repository root.
buildbot-nix reads it from the default branch when discovering projects and from the PR branch when evaluating pull requests (except for `effects_on_pull_requests`, which is always read from the default branch for security).

`attribute` (string, default `"checks"`) scopes which flake output nix-eval-jobs evaluates.
The default restricts evaluation to the `checks` flake output.
When set to a different value, only the specified attribute path is traversed.

`effects_on_pull_requests` (boolean, default `false`) controls whether effects discovered in a PR's flake outputs are executed.
Defaults to false for security: PR authors could otherwise define arbitrary effects that execute with the buildbot worker's credentials.
This field is always read from the default branch's `buildbot-nix.toml`, not from the PR branch, preventing the attack where a PR enables its own effects.

`effects_branches` (list of glob patterns, optional) specifies additional branches beyond the default branch whose pushes trigger effect execution.
By default, effects run only on pushes to the default branch.
Adding `["release/*", "deploy/*"]` enables effects on matching branch pushes.


## Attribute scoping patterns

The `attribute` field determines the evaluation scope and directly affects CI reliability.

`checks.x86_64-linux` is the most common setting for repositories that target linux.
It restricts evaluation to checks for the worker's system, avoiding failures from cross-system attributes (e.g., `checks.aarch64-darwin`) that the worker cannot evaluate or build.

Omitting `attribute` uses the default value `"checks"`, restricting evaluation to the `checks` flake output.
To evaluate additional outputs, set `attribute` to a broader path.

Multi-system scoping is not directly supported by a single `attribute` value.
Repositories targeting multiple systems need either multiple workers (one per system) or a CI strategy that runs nix-fast-build separately per system in a workflow matrix.

For repositories using `perSystem` from flake-parts, the attribute path reflects the flake-parts output structure.
`checks.x86_64-linux` includes all `perSystem` checks for that system.


## `fullyPrivate` mode

The `fullyPrivate` option on the buildbot-nix NixOS module adds oauth2-proxy in front of the buildbot web UI, requiring OAuth2 authentication for all web access.
It forces `authBackend` to `httpbasicauth`, which makes `github.enable` default to `false`.
To retain GitHub commit status posting under `fullyPrivate`, set `github.enable = true` explicitly.

This mode is appropriate for organizations where the buildbot instance should require OAuth2 authentication for web access, such as instances accessible over a VPN (e.g., ZeroTier mesh).
It is also useful during initial setup when the web UI should not be publicly accessible without authentication.

The web UI requires OAuth2 authentication under `fullyPrivate` mode.
Users authenticate via the configured OAuth2 provider (e.g., GitHub, Gitea) before accessing the buildbot dashboard.


## GitHub App configuration

buildbot-nix uses a GitHub App for repository access and status reporting.
The App requires the following permissions:

- `checks:write` for posting check run results
- `statuses:write` for posting commit status updates
- `contents:read` for cloning repositories
- `pull_requests:read` for discovering PR events and reading PR metadata

The `email` permission is additionally required when using `fullyPrivate` mode with oauth2-proxy, whose hardcoded OAuth2 scope includes `user:email`.
Missing this permission in `fullyPrivate` mode causes silent failures in OAuth2 authentication.
This is a common source of confusion during initial setup of private buildbot instances.

The GitHub App's webhook URL must point to the buildbot instance's webhook endpoint.
For dual-zone network architectures (public internet for webhooks, VPN for admin), the webhook endpoint must be publicly accessible even if the buildbot web UI is VPN-only.


## Worker configuration via clan vars

The `buildbot-nix-worker` clan vars generator creates both the worker password and `workers.json` atomically.
This ensures the master and worker share the same credentials without manual synchronization.

`workers.json` specifies the workers available to the buildbot master.
Each entry includes:

- `name`: worker identifier
- `pass`: shared secret (generated by the vars generator)
- `cores`: number of parallel build jobs the worker supports

The `cores` value must match the worker machine's `nproc` output.
For a Hetzner CX53 with 8 shared vCPU and SMT (simultaneous multithreading), `nproc` reports 16.
Setting `cores` to 16 allows the buildbot master to schedule up to 16 parallel derivation builds on that worker.
Underprovisioning (setting cores lower than nproc) wastes available parallelism.
Overprovisioning (setting cores higher) causes thrashing as build jobs compete for insufficient CPU time.

The vars generator produces two secrets: the worker password file and the `workers.json` file.
The master's NixOS module references both via `systemd.services.buildbot-master.serviceConfig.LoadCredential`.
The worker's NixOS module references only the password file.


## Dual-scheduler architecture

buildbot-nix's internal scheduling separates evaluation from building.

The first scheduler, triggered by push or PR events, runs nix-eval-jobs against the configured attribute scope.
This produces a list of derivations to build.
The evaluation result is reported as the `buildbot/nix-eval` status check.
Evaluation failures (nix syntax errors, infinite recursion, missing inputs, import failures) are reported at this stage without triggering any builds.

On evaluation success, the second scheduler creates per-attribute build jobs.
Each derivation discovered by nix-eval-jobs becomes an independent build.
Individual build results are aggregated into the `buildbot/nix-build` status check.
A build failure in one attribute does not prevent other attributes from building.

This separation has practical consequences for debugging.
A `buildbot/nix-eval` failure means the flake does not evaluate cleanly.
The developer should run `nix flake check --dry-run` locally to reproduce the evaluation error.
A `buildbot/nix-build` failure means one or more derivations failed to build.
The developer can build the specific failing attribute locally: `nix build .#checks.x86_64-linux.failing-check`.


## Effect pipeline configuration

Effects discovered by nix-eval-jobs are executed after all checks pass.
The pipeline is sequential: evaluate, build checks, discover effects, run effects.
Effects do not run in parallel with checks.

`perRepoSecretFiles` on the buildbot-nix master NixOS module maps repository identifiers to secret JSON files.
The mapping supports exact repo matches and organization-wide wildcards:

```nix
perRepoSecretFiles = {
  "github:org/specific-repo" = "/run/credentials/specific-repo-secrets.json";
  "github:org/*" = "/run/credentials/org-default-secrets.json";
};
```

Exact repo matches take priority over wildcard patterns.
Secret files are loaded via systemd `LoadCredential` and mounted into the bubblewrap sandbox at `/run/secrets.json` when executing the effect.

The `effects_on_pull_requests` safety default (`false`) means effects never run on PR pushes unless explicitly enabled in the default branch's configuration.
This prevents PR authors from defining effects that execute with the buildbot worker's credentials.
Even when enabled, effects from PRs use the same secret mapping as the default branch, so PR authors cannot inject custom secret references.

Scheduled effects are discovered when a push to the default branch includes `onSchedule` entries in the `herculesCI` flake output.
Each schedule produces a Buildbot Nightly scheduler.
Changing the schedule in the flake and pushing to the default branch triggers a buildbot reconfig that updates the Nightly scheduler's timing.


## Mergify gating

Merge automation gates on both buildbot status checks and any remaining GitHub Actions checks.
The typical mergify configuration for a repository that has completed the phase 2 migration:

```yaml
pull_request_rules:
  - name: merge when all checks pass
    conditions:
      - check-success=buildbot/nix-eval
      - check-success=buildbot/nix-build
      - check-success=check-fast-forward
      - "#approved-reviews-by>=1"
    actions:
      merge:
        method: merge
```

The `check-fast-forward` condition is a GitHub Actions check that verifies the PR can be fast-forwarded to main.
This is separate from buildbot-nix and remains in GitHub Actions because it is a git operation.

Repositories in the phase 1 to phase 2 transition period may have both `check-fast-forward` and a GitHub Actions nix-check job listed alongside the buildbot checks.
Once the nix-check job is removed in phase 2 cleanup, only the buildbot checks and fast-forward check remain.

Additional conditions (review approval counts, label requirements, branch protection rules) are orthogonal to the buildbot integration and configured per-repository as needed.
