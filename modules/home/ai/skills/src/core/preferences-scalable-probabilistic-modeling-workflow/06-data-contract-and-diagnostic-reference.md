# Data contract and diagnostic reference

This section specifies how probabilistic modeling artifacts interface with persistent storage and in-memory interchange, establishes ArviZ as the conceptual vocabulary for inference diagnostics, and delineates which quantitative diagnostics apply to which inference paradigm.

## Data contract with the Arrow-native data layer

The probabilistic modeling workflow produces and consumes tabular artifacts at every phase.
These artifacts follow the Arrow-native data layer conventions defined in `preferences-data-modeling`, using Arrow columnar storage for persistence, Arrow record batches for in-memory interchange, and DuckDB for queryable access.
The contract is language-agnostic: DuckDB.jl, duckdb (Python), and duckdb-rs all produce Arrow record batches through the same C Data Interface.

### Persistent artifacts

All durable artifacts are stored in Arrow-native columnar formats (Parquet today, Vortex or Lance as they mature) and are queryable via DuckDB.

Simulation ensembles (Step 6) are the largest artifacts, comprising millions of (theta, y) pairs generated by ancestral sampling from the prior predictive distribution.
Arrow list and struct columns accommodate variable-length trajectories, irregular observation grids, and heterogeneous state vector shapes within a single table.
Offline ensembles are persistent and reusable across approximator retraining attempts, subject to the invalidation rules in section 05: changing the prior, dynamical model, or observation model invalidates the ensemble, while changing only the approximation architecture allows reuse.

Diagnostic tables (Steps 10-14) store the quantitative outputs of algorithmic and inferential calibration.
These include SBC ranks, z-scores, contractions, NRMSE values, and calibration metrics, organized with one row per (dataset_id, parameter) pair.
DuckDB tables serve these directly for interactive querying and aggregation.

Posterior samples (Step 12) are stored in tabular form with columns for sample_id and each parameter.
This layout supports direct computation of summary statistics, credible intervals, and pairwise correlations via standard SQL or DuckDB analytical functions.

Model provenance registries track the relationship between model versions and their derived artifacts.
Each row records (model_version, ensemble_path, approximator_path, timestamp, invalidated_by), linking the simulation ensemble and trained approximator back to the specific generative model that produced them.
This registry operationalizes the invalidation rules from section 05, making it queryable which artifacts remain valid after a model change.

Experimental design sweep results (Step 11) store the outputs of computational experimental design, with columns for design_config, parameter, and metric_value.
These tables record how inferential adequacy varies across observation designs (number of time points, observed variables, noise levels), enabling data-driven experimental planning.

### In-memory interchange

Arrow record batches serve as the interchange format between query operations and computational operations.
DuckDB query results materialize as Arrow record batches, which flow to downstream computation without serialization overhead.
Arrow IPC provides the streaming protocol when data moves between processes or across language boundaries within a single analytical pipeline.

### Adapter boundary

The boundary between the data layer and the computational inference layer is explicit.
Arrow record batches are the input; framework-native tensors (NumPy arrays, JAX arrays, PyTorch tensors, or Julia arrays) are the output.
The adapter converts at the training boundary, handling dtype conversions, log-transforms of positive quantities, normalization, and reshaping into the tensor layouts expected by the neural approximator or sampler.
This boundary corresponds to the preprocessing pipeline described in Step 8(c).

### GPU streaming

Data flows to GPU memory via Arrow IPC or framework-native tensor transfer, depending on the computational backend.
The choice is determined by the inference framework's data loading conventions rather than prescribed here.

### Web serving

DuckDB over httpfs with predicate pushdown serves diagnostics and results to visualization tools.
This enables interactive exploration of diagnostic tables, posterior samples, and experimental design sweeps without requiring the full computational environment to be active.

### Cross-references

See `preferences-data-modeling` for Arrow substrate details, cross-language bindings, DuckLake operational patterns, columnar format evolution, and scientific data contracts.
The conventions defined there apply to all artifacts described in this section.

## ArviZ as conceptual reference

ArviZ (Python) and Arviz.jl (Julia) define the standard vocabulary and schema structure for organizing Bayesian inference artifacts.
Their InferenceData schema provides logical groupings that serve as the reference taxonomy for how inference outputs are structured, regardless of the physical storage format used.

The relevant schema groups are:

- **posterior**: parameter samples from the fitted posterior distribution
- **prior**: parameter samples from the prior distribution
- **prior_predictive**: simulated observations drawn from the prior predictive distribution
- **posterior_predictive**: simulated observations drawn from the posterior predictive distribution
- **observed_data**: the actual observed dataset used for inference
- **log_likelihood**: pointwise log-likelihood values for model comparison
- **sample_stats**: metadata about the inference process (acceptance rates, step sizes, divergences for MCMC; training loss, validation loss for amortized methods)

These groups map onto the workflow steps: prior and prior_predictive arise from Steps 6-7, posterior and posterior_predictive from Steps 12-14, and sample_stats from Steps 9-10.

The coupling to ArviZ is conceptual, not mechanical.
ArviZ's native storage format is NetCDF backed by xarray, which is not part of our data layer.
The Arrow-native stack described above replaces both: Parquet files replace NetCDF for persistence, DuckDB tables replace xarray for structured access, and Arrow record batches replace in-memory xarray datasets for interchange.

ArviZ's plotting conventions (trace plots, pair plots, forest plots, rank plots, ECDF plots, posterior predictive check overlays) define the standard visual vocabulary for Bayesian diagnostics.
Implementation of these visualizations should use the Arrow-native stack for data access, but the plots themselves follow ArviZ's conventions for layout, labeling, and interpretation.

The framing is: ArviZ defines *what* to compute and visualize; our Arrow-native stack defines *how* to store and serve it.

## Diagnostic threshold delineation

Different inference paradigms require fundamentally different diagnostics.
Applying chain-convergence diagnostics to amortized inference, or approximator-calibration diagnostics to MCMC, is a category error.
The following delineation clarifies which diagnostics belong to which paradigm and which are shared.

### Sampling-based inference (MCMC, HMC, VI)

These diagnostics assess chain convergence and mixing quality.
They are irrelevant to amortized simulation-based inference because there are no chains to diagnose.

- R-hat <= 1.01: potential scale reduction factor, measuring convergence across independent chains
- ESS >= 100 x num_chains: effective sample size, measuring mixing efficiency relative to total samples drawn
- Zero divergences: specific to Hamiltonian Monte Carlo, indicating the numerical integrator has not encountered pathological posterior geometry
- E-BFMI > 0.3: energy Bayesian fraction of missing information, detecting poor exploration of the typical set in HMC

### Simulation-based inference (amortized NPE, NLE, NRE)

These diagnostics assess approximator calibration and recovery quality.
They are irrelevant to sampling-based inference because there is no trained approximator to calibrate.

- SBC rank uniformity via log-gamma statistic (Modrak et al. 2025, *Bayesian Analysis*): a quantitative test for rank uniformity that is more sensitive than visual ECDF inspection, providing a scalar test statistic with known null distribution
- Calibration ECDFs with simultaneous confidence bands: continuous generalization of SBC rank histograms that reveal the pattern of miscalibration (over-concentration, under-concentration, bias) across the full probability range
- Recovery plots: scatter of posterior mean versus true parameter value across the simulated ensemble, with ideal recovery falling along the identity line
- Z-score and contraction: posterior z-score quantifies standardized distance from truth, contraction quantifies posterior shrinkage relative to prior, and their joint distribution diagnoses non-identifiability, overfitting, and computational artifacts (see section 01)
- NRMSE: normalized root mean squared error of posterior means across the ensemble, providing a single scalar measure of point-estimate accuracy
- Model misspecification detection via summary-space MMD (Schmitt et al. 2021): maximum mean discrepancy between real and simulated data in the learned summary space, detecting when the generative model fails to reproduce structural features of the observed data
- Ensemble inference: training multiple networks with controlled data overlap quantifies epistemic uncertainty about the approximation itself, which is distinct from the posterior uncertainty about parameters; disagreement between ensemble members indicates regions where the training data is insufficient

### Both paradigms

These diagnostics apply regardless of the inference method, addressing model adequacy rather than computational faithfulness.

Prior predictive checks (Step 5) verify that the prior generates plausible observations according to domain expertise thresholds.
This is a property of the generative model, independent of how inference will be performed.

Posterior retrodictive checks (Step 14) verify that posterior samples reproduce observed data patterns when fed back through the simulator.
The check requires re-simulation regardless of whether posterior samples came from MCMC chains or a neural approximator.

LOO-CV and model comparison operate in both paradigms but with different implementations.
Sampling-based inference uses PSIS-LOO (Pareto-smoothed importance sampling leave-one-out cross-validation).
Simulation-based inference uses C2ST (classifier two-sample test) or the log-gamma statistic for calibration-based comparison.
Amortized model comparison can itself be amortized by training a model-selection network that maps observations directly to model posterior probabilities.

Out-of-distribution detection (Step 13) checks whether the observation falls within the support region where inference is validated.
For sampling-based methods, this manifests as divergences or poor mixing in regions the sampler has not been tuned for.
For amortized methods, this manifests as the observation's learned summary falling outside the training distribution in summary space.
