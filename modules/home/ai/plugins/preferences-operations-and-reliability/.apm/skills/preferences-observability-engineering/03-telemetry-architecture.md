# Telemetry architecture

## Trace structure

A trace is a directed acyclic graph of span events representing the path of a single request through the system.
Each span records a unit of work: handling an HTTP request, executing a database query, calling a downstream service.
The graph structure captures the causal relationships between these units of work.

Five fields are required per span.
The trace_id is generated by the root span (the first span in the trace) and propagated to every downstream span.
It is the identifier that ties all spans in a trace together.
The span_id is unique to each span and distinguishes it from all other spans in the trace.
The parent_id references the span_id of the caller; it is absent in the root span.
The timestamp records when the span began.
The duration records how long the span lasted.

Additional enrichment makes spans useful for investigation.
The service name identifies which service produced the span.
The span name describes the operation (e.g., "GET /api/users", "SELECT users", "publish order.created").
Arbitrary key-value attributes carry the business context described in `02-instrumentation-patterns.md`: user IDs, tenant IDs, feature flag states, cache hit status, and any other dimension relevant to debugging.

The root span generates the trace_id.
Every downstream service inherits the trace_id via context propagation, creates its own span_id, and sets its parent_id to the caller's span_id.
The result is a tree (or DAG, if a service fans out to multiple concurrent downstream calls) that reconstructs the full request path.

## Context propagation

Trace context flows between services via HTTP headers, message metadata, or other transport-level mechanisms.
The W3C TraceContext specification defines the standard header format (`traceparent` and `tracestate`); B3 format is an older alternative still in use.
Context propagation is the connective tissue that stitches independently produced spans into coherent traces across service boundaries.

The propagation mechanism is general: any correlated set of events can be stitched into traces using the same trace_id mechanism.
Batch job stages can share a trace_id, producing a trace that shows the progression through pipeline steps.
CI pipeline phases can share a trace_id, producing a trace that shows build, test, and deploy stages with their durations and outcomes.
Serverless function chains can share a trace_id, producing a trace that shows event-driven propagation across functions.
The requirement is only that the trace_id is propagated to each participant and that each participant creates spans with the appropriate parent_id.

## Algebraic properties of traces

The trace DAG exhibits algebraic structure that illuminates its compositional properties.

The trace is a free category generated by the call graph.
Each span is a morphism from its start state to its end state.
The trace_id is the identity of the composite (the trace as a whole).
The span_id is the identity of each morphism (the span as an individual unit of work).
The parent_id establishes composition: span B with parent_id pointing to span A means B composes with A (B follows A in the causal chain).

Context propagation exhibits comonadic structure.
Each span inherits the cumulative context of all its ancestors: the extract operation gives the current span's data, and the duplicate operation gives the span in its full surrounding trace context.
This comonadic structure means that every span carries not just its own data but the entire causal history that led to it, enabling investigation that starts at any span and traces backward through the ancestry to understand how the system arrived at that state.

## Telemetry pipeline architecture

As systems grow in complexity, the many-to-many relationship between telemetry producers (services) and telemetry consumers (backends, analysis tools, compliance systems) requires an intermediary.
The telemetry pipeline provides this intermediary, abstracting the routing, transformation, and delivery of telemetry data.

A pipeline consists of four stages: receiver, buffer, processor, and exporter.
The receiver accepts telemetry in one or more formats (OTLP, proprietary formats, legacy protocols).
The buffer provides resilience against downstream outages and traffic spikes.
The processor transforms telemetry: filtering, sampling, enriching, redacting, and routing.
The exporter delivers telemetry to one or more backends in the format each backend expects.

Pipeline stages compose sequentially.
A receiver feeds into a buffer, which feeds into one or more processors, which feed into one or more exporters.
Complex pipelines may branch: the same input flows to different processor chains depending on signal type, producing different outputs for different backends.

The pipeline serves several functions beyond routing.
Security and compliance requirements may demand PII redaction or GDPR-compliant data handling before telemetry leaves the network boundary.
Workload isolation separates high-volume telemetry (request traces from a busy service) from low-volume telemetry (audit logs) to prevent noisy producers from impacting critical signals.
Data buffering (often backed by a durable queue) provides resilience when backends are temporarily unavailable.
Capacity management applies rate limiting, sampling, and queuing to prevent telemetry volume from overwhelming backends or the pipeline itself.
Data transformation normalizes formats and schemas across heterogeneous producers.
Data quality enforcement rejects or flags telemetry that does not conform to organizational schemas.

When a common envelope format is used internally, it acts as a universal intermediate representation.
All inputs are mapped into the envelope on receipt; all outputs are mapped from the envelope on export.
This decouples the input format matrix from the output format matrix, reducing the integration surface from M-times-N to M-plus-N.

## Sampling strategies

At production scale, emitting and storing every trace is neither economically feasible nor analytically necessary.
Sampling reduces volume while preserving the ability to investigate system behavior.

Head-based sampling makes the keep-or-drop decision at the trace root, before any downstream work occurs.
The decision is propagated to all downstream spans via the trace context, ensuring that all spans in a kept trace are kept and all spans in a dropped trace are dropped.
Head-based sampling is simple to implement and produces complete traces, but it cannot select traces based on outcome: an interesting trace (one with errors, high latency, or unusual behavior) is equally likely to be dropped as a boring one.

Tail-based sampling makes the decision after the trace completes, based on observed properties.
It can prioritize interesting traces: keep all error traces, keep all traces above a latency threshold, keep traces that touch a specific service.
Tail-based sampling produces a biased sample that over-represents interesting behavior, which is exactly what debugging requires.
The cost is complexity: the sampling system must buffer complete traces before deciding, which requires memory and introduces latency.

Consistent sampling uses a deterministic hash of the trace_id to make the sampling decision.
Every service that sees the same trace_id makes the same decision independently, without coordination.
This ensures that traces are kept or discarded atomically: partial traces (where some spans were kept and others dropped) are useless for investigation and waste storage.

## Sample rate as Radon-Nikodym derivative

Each sampled event carries a `sample_rate` field recording the inverse probability of its inclusion.
An event sampled at 1-in-10 carries a sample_rate of 10, meaning it represents 10 events in the original population.

Aggregation over sampled data must use weighted sums.
COUNT is the sum of sample_rates across all kept events.
SUM of a field x is the sum of x multiplied by sample_rate for each kept event.
AVERAGE of x is the weighted sum divided by the weighted count.

The sample_rate functions as a Radon-Nikodym derivative: it is the density of the original measure with respect to the sampled measure.
This allows reconstruction of statistical properties of the original distribution from the sampled subset, provided the sample is consistent (every event had a well-defined inclusion probability).
Without sample_rate annotations, aggregations over sampled data are biased and the bias is unquantifiable.

## Cardinality management

High-cardinality fields are the most valuable for debugging.
A user ID, request ID, or session token uniquely identifies one entity, enabling the investigator to follow that entity through the system.
Low-cardinality fields (HTTP method, status code category, datacenter region) are useful for grouping and filtering but insufficient for root cause analysis of individual-entity problems.

Metrics-based systems cannot handle high cardinality at scale.
Metrics systems pre-aggregate data into time series, where each unique combination of label values creates a separate series.
A metric with a user_id label containing one million unique values creates one million time series, exceeding the cardinality limits of most metrics backends.
This is a categorical limitation of the metrics data model, not an engineering trade-off that better implementation could resolve.

Event-based observability backends store raw events and compute aggregations at query time.
High-cardinality fields are stored as attributes on individual events and are available for filtering, grouping, and investigation without pre-aggregation.
The cost model differs: metrics systems pay at write time (one series per cardinality combination), event systems pay at query time (scanning and aggregating raw events).
For debugging, where queries are ad hoc and the cardinality of interest is unpredictable, the event model is fundamentally better suited.

The practical implication is that attempts to use metrics systems for debugging (adding high-cardinality labels to metrics to answer debugging questions) will eventually hit cardinality limits.
The correct architecture routes debugging signals to an event-based backend and uses metrics for known, low-cardinality operational indicators.

## Column store properties

Observability backends typically store events in column-oriented format.
Each event is a product type: a record with named fields of various types.
The column store decomposes this product type into its constituent fields (projections), storing all values of each field contiguously.

Queries reconstruct the relevant sub-product by joining columns on row index.
A query that filters on three fields and returns five fields only reads eight columns, regardless of how many total columns the event contains.
This property makes wide events (300-400 fields) practical: the storage cost is proportional to the total data volume, but the query cost is proportional to the number of fields touched by the query.

Segments are indexed by time windows, enabling parallel scan-filter-aggregate-reduce across time ranges.
A query over the last hour reads only the segments covering that hour, not the entire dataset.
Within each segment, column compression exploits the statistical properties of each field independently: low-cardinality fields compress extremely well, timestamps compress with delta encoding, and even high-cardinality fields benefit from dictionary encoding.

The column store architecture is what makes the core analysis loop practical at scale.
The dimension diff operation (comparing all field values between two populations) requires reading every column, but only for the rows in the two populations.
Time-range filtering narrows the rows; column projection narrows the fields.
The combination keeps query costs manageable even on datasets with billions of events and hundreds of dimensions.

## Retention and tiering

Not all telemetry needs the same retention period or query performance.
Recent data (last 24-72 hours) is queried frequently during active debugging and benefits from fast storage with low query latency.
Historical data (weeks to months) is queried occasionally for trend analysis, capacity planning, and retrospective investigation.
Archival data (months to years) is retained for compliance, audit, or long-term trend analysis and is queried rarely.

Telemetry pipelines and backends can implement tiering strategies that move data through storage tiers as it ages.
Hot storage provides low-latency queries over recent data.
Warm storage provides acceptable query latency at lower cost for historical data.
Cold storage provides archival retention at minimal cost with higher query latency.

The retention policy should be informed by the SLO window.
If the error budget uses a 30-day sliding window, the observability backend must retain at least 30 days of data at query-able granularity to support investigation of any burn event within the window.
Sampling rates can differ by tier: keep full fidelity for the hot tier (recent data) and increase the sampling rate for warm and cold tiers.
