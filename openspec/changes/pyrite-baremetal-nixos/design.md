## Context

pyrite is an Apple MacBookPro14,1 (2017 13-inch, non-Touch-Bar) that would become the vanixiets fleet's first bare-metal NixOS machine.
The fleet today is four nix-darwin laptops and five NixOS cloud VMs, and every NixOS machine ever created in this repository is a cloud VM.
Every hardware assumption in the shared `base` layer is consequently untested on physical hardware, and several of them are cloud-VM assumptions that do not hold here.

Live reconnaissance on 2026-07-16 (Linear CAM-31, complete) established the hardware surface from the running machine over ssh while it was booted from a stock NixOS 26.05 installer ISO.
`docs/notes/development/hardware/pyrite-hardware-inventory.md` records what was observed; the raw dumps sit beside it.
`docs/notes/development/hardware/mbp141-nixos-clan-research.md` records the prior inference-based research, and its own provenance header names the three places the live recon supersedes it.

This change implements CAM-32.
It adds the machine, and it adds the bare-metal install path, because the repository presently has no install path that does not route through terranix.

## Adversarial verdicts that change the design

Several load-bearing claims went to adversarial verification and were refuted.
They are recorded here rather than designed around silently, because they alter the justification for decisions the user has already made and one of them alters the ordering of the apply.

### Refuted: "ZFS is fleet convention because base forces forceImportRoot"

This was the premise the ZFS decision was originally stated to rest on, and as an argument it does not hold.

`modules/system/zfs-force-import.nix:19-21` does set `boot.zfs.forceImportRoot = true` on `flake.modules.nixos.base`, and all five NixOS machines import `base`, so the setting is real.
But the inference from that setting to "ZFS is convention" is invalid.
The option is declared unconditionally in nixpkgs, keyed on `stateVersion` alone with no reference to whether ZFS is enabled, and it is inert on a machine with no pool.
clan-core demonstrates the distinction structurally: `nixosModules/clanCore/zfs.nix` sets `boot.zfs.forceImportRoot = lib.mkDefault false` unconditionally while gating `services.zfs` behind `lib.mkIf config.boot.zfs.enabled` in adjacent lines.
The setting is a defensive default and proves nothing about ZFS usage.

The conclusion survives on evidence the original argument never cited.
All five NixOS machines root on ZFS via disko with a `zroot` pool and datasets `root`, `root/nixos`, `root/home`, `root/nix`; none has a non-ZFS root.
ZFS-on-root is genuinely the convention, and the decision stands — but it is a filesystem argument, not an argument about `forceImportRoot`.

### Refuted: "ZFS native encryption requires a key pushed at every boot by a second machine"

This is the false premise on which an earlier revision of this document recommended LUKS, and it is the reason this design was rewritten.

The premise came from reading clan-infra's `web01`, whose `machines/web01/disko.nix:96` sets `keylocation = "file://${config.clan.core.vars.generators.zfs.files.key.path}"` and keeps that value at boot, backed by an initrd unit at `disko.nix:63-67` that spins waiting for the key file to appear.
Generalizing from that one machine produced the conclusion that ZFS native encryption structurally cannot use a boot-time prompt under a non-interactive install, and therefore that LUKS — whose `cryptsetup luksFormat` reads a `passwordFile` without any interactive phase — was the only mechanism satisfying both re-runnability and a typed passphrase.

The generalization is wrong, and disko refutes it directly.
`lib/types/zfs_fs.nix:80-91` enumerates the properties ZFS treats as `PROP_ONETIME` — `encryption`, `casesensitivity`, `utf8only`, `normalization`, `volblocksize`, `pbkdf2iters`, `pbkdf2salt`, `keyformat`.
`keylocation` is deliberately absent from that list, which is what makes it settable after creation.
disko then documents the resulting idiom in its own encrypted-root example at `example/zfs.nix:104-113`: create the dataset with `keylocation = "file:///tmp/secret.key"`, and flip it with a `postCreateHook` running `zfs set keylocation="prompt"`.
`postCreateHook` is a real option declared at `lib/default.nix:476` and spliced after the create body at `lib/default.nix:507-511`, so the flip is ordinary supported disko.

The key material never changes across that flip; only the location from which ZFS reads it does.
This is the exact ZFS-native analog of LUKS's `passwordFile`, and it dissolves the only advantage LUKS had.
`zfs change-key` is not involved, is not needed, and appears nowhere in clan-core, clan-infra, or Mic92's dotfiles.

web01 is a server that is unlocked remotely by another machine, so it has no reason to flip the property and does not.
Reading pyrite's requirements off web01's configuration was the error.

### Refuted: "facter.json can be placed exactly like the other five, with no special casing"

Two of the three conjuncts fail, and the first one changes the apply ordering.

The placement path is symmetric and that part holds: `machines/pyrite/facter.json`, git-tracked, no import line, no flake input, no facter binary.
clan-core wires it by path existence at `nixosModules/clanCore/nixos-facter.nix:5,10`, and `hardware.facter` itself is upstreamed into nixpkgs rather than supplied by an input.

But `machines/pyrite/` is not an inert data directory.
clan-core `modules/clan/module.nix:141-148` `readDir`-scans `${directory}/machines` and injects an inventory machine per subdirectory, with `machineClass` defaulting to `"nixos"`, from which `nixosConfigurations` is filtered.
Creating the directory therefore materializes an inventory entry and a `nixosConfigurations.pyrite` before any module exists to configure it, breaking both hardcoded name lists in `modules/checks/structure/flake-shape.nix` and emitting a `checks.nixos-pyrite` that builds a machine with no filesystems and no boot loader.
The consequence is an ordering constraint recorded in tasks.md: the facter report lands in the same commit as the module and the registrations, never before them.
This is why the report is staged inside this change directory rather than in `machines/` while the change is in flight.

The report is also a superset, not a match.
pyrite's carries a `uefi` key the five lack, and reports `virtualisation = "none"` against their `kvm` and `google`.
Three nixpkgs facter code paths that are dead on every existing machine become live: `hardware/facter/boot.nix` reads `uefi.supported` and sets `boot.loader.grub.efiSupport` (inert under systemd-boot); `hardware/facter/firmware.nix` gates its entire block on bare-metal detection and newly sets `hardware.enableRedistributableFirmware` and `hardware.cpu.intel.updateMicrocode`, both `mkDefault`; and `hardware/facter/networking/initrd.nix` injects `brcmfmac` into `boot.initrd.kernelModules` because `base` sets `boot.initrd.network.enable = true`.
None of these is harmful, but they are unreviewed defaults no one in this fleet has evaluated, and the design asserts what it wants rather than inheriting them.

Note what facter does not supply: `hardware/facter/keyboard.nix` sources initrd keyboard modules from the USB controller report only, and `applespi` is SPI.
The stage-1 passphrase prompt gains nothing from facter and must carry its own modules.

### Refuted: "removing terranix leaves no functional gap other than a disko layout"

The dependency half of this claim holds and is worth stating, because it makes the change smaller than it looks.
terranix supplies pyrite no dependency: no DNS, no IP, no ssh host key, no cloud-init, no bootstrap secret, no `targetHost` value, and nothing the clan inventory or any flake check reads.
Three NixOS machines already run with `enabled = false` terranix entries whose cloud resources do not exist, so a machine with no entry at all evaluates and checks clean.
pyrite needs no terranix entry.

The capability half fails, and its carve-out names the wrong thing.
terranix never supplied disko — disko has always lived at `modules/machines/nixos/<name>/disko.nix`.
What terranix does supply, and pyrite needs an analog of, is two things the claim omits: the repository's only recorded `clan machines install` invocation, and install-time SSH credential material with root-login enablement.
There is no `clan machines install` recipe in the justfile and no other invocation site anywhere.
Taken at face value, this claim would ship a machine module with a disko layout and no recorded install path — a one-off manual install, which the binding decision rules unacceptable.
D8 exists because of this.

### Refuted: "tagging `nixos` in the inventory auto-enrolls the machine as a Tor relay"

The mechanism is real; the characterization is wrong, and the correction matters because it changes what is being decided.

`modules/clan/inventory/services/tor.nix` does target `roles.server.tags."nixos"`, and `modules/clan/inventory/services/sshd.nix` targets the same tag for both its server and client roles, so the tag that supplies sshd host keys and CA certificates does also select the tor server role.
That much of the claim holds.

But clan-core's tor server role is not a relay.
`clanServices/tor/default.nix` describes the role as "Sets up a Tor onion service for the machine, thus making it reachable over Tor", and its `nixosModule` sets only `services.tor.enable = true` and `services.tor.relay.onionServices."clan_<instance>"`, with a default `portMapping` exposing port 22.
In nixpkgs, `services.tor.relay.enable` — "Whether to enable relaying of Tor traffic for others" — is a separate `mkEnableOption` at `nixos/modules/services/security/tor.nix:531-545`, defaulting false, and it gates the `ExitPolicy`/`BridgeRelay` settings at `tor.nix:1277-1296`.
The `relay.onionServices` attribute lives under the `relay` namespace but does not enable relaying; nothing forwards other people's traffic.

What the tag actually produces on pyrite is a continuously running tor daemon publishing a v3 onion service that exposes the machine's sshd to the Tor network, plus a `tor_tor` vars generator holding the onion secret key.
That is a real cost, and it is not the one the claim named — so what to do about it does not follow from the correction, and is decided on the merits rather than by the relay characterization.
See D10.

### Confirmed: the stage-1 passphrase prompt works on the internal keyboard

This is the safety-critical claim, because a wrong answer means being locked out of a freshly-wiped laptop with no macOS to fall back to.
It was verified end to end and it holds, with one mechanism in the earlier write-up corrected below.

The nixos-hardware profile at `apple/macbook-pro/14-1/default.nix:19-24` sets `boot.initrd.kernelModules` — the strong option that force-loads, not merely `availableKernelModules` — to `applespi`, `spi_pxa2xx_platform`, `intel_lpss_pci`, and `applesmc`, under a comment at `:15-16` reading "Make the keyboard work in stage1".
The module closure was built at this repository's pinned nixpkgs (kernel 6.18.37, not the ISO's 7.1.3) with `allowMissing = false`, which exits non-zero on any unresolved root module, and it succeeded — producing `applespi.ko.xz`, `spi-pxa2xx-platform.ko.xz`, `spi-pxa2xx-core.ko.xz`, `intel-lpss.ko.xz`, `intel-lpss-pci.ko.xz`, and `applesmc.ko.xz`, matching the set recon observed running.

The prompt-display mechanism, stated precisely.
`nixos/modules/tasks/filesystems/zfs.nix:159-165` gives the `zfs-import-zroot` unit `after = [ "systemd-modules-load.service" "systemd-ask-password-console.service" ]`, and `zfs.nix:227` runs `systemd-ask-password --timeout=${passwordTimeout}` with the option defaulting to an unbounded wait.
The `systemd-modules-load.service` half of that ordering is real and load-bearing: the unit is in the initrd transaction, so the SPI modules are modprobed before the prompt is issued.
The `systemd-ask-password-console.service` half is vacuous.
That service is not in the boot transaction; it is triggered by `systemd-ask-password-console.path`, whose `[Path] DirectoryNotEmpty=/run/systemd/ask-password` is level-triggered, so the agent starts when the password request appears regardless of ordering.
`nixos/modules/system/boot/systemd/initrd.nix:96-97` includes both the `.path` and the `.service` in the initrd unit set, which is what actually makes the prompt render.
Nothing in the design depends on the vacuous edge, but the earlier statement that it orders the prompt was wrong and is corrected here.

`i915` reaches initrd via the profile's import chain (`apple/macbook-pro/14-1` imports `common/cpu/intel/kaby-lake`, which reaches `common/gpu/intel`, whose `default.nix:90` adds `cfg.driver` to `boot.initrd.kernelModules` when `loadInInitrd`), providing the framebuffer console that displays the prompt.
`i915` is not the profile's alone, however: the committed facter report supplies it too, through a graphics module `nixos/modules/module-list.nix:70` imports into every NixOS configuration unconditionally.
What rests on the profile alone is the SPI input path, which is why D5 declines the "skip the profile" fallback on those modules rather than on this one.

Two residual honesties.
The modules are proven shipped and loaded; that `applespi` binds and yields working input at 6.18.37 in initrd specifically is a strong inference from recon's observation at 7.1.3 in a full system, not direct evidence.
And `systemd-modules-load` returns when `modprobe` returns, not when the input device exists, so the prompt can appear seconds before the keyboard is live — mitigated by the unbounded timeout, and not a lockout.
A not-yet-live keyboard submits nothing and therefore burns no attempt, because `--timeout=0` blocks until Enter rather than returning empty.

Wrong answers are capped, however.
`zfs.nix:224-231` wraps the prompt in a `tries=3` loop and gates the unit on `[[ $success = true ]]`, so a third consecutive mistyped passphrase fails `zfs-import-zroot` and drops the boot to an emergency shell.
That is a reboot and a retry rather than a lockout — the pool is untouched and the key is unchanged — but it is a plausible outcome for a roughly 45-character random passphrase typed blind, and it is the reason D4's generator emits words rather than hex.

Lockout is in any case not the intended failure mode: `usbhid`, `hid-generic`, and `hid-apple` reach the initrd through `availableKernelModules` and udev autoloading rather than through the force-loading `kernelModules`, so an external keyboard is a recovery path that depends on udev probing rather than on an unconditional modprobe.
The runbook carries this, along with the fact that a MacBookPro14,1 has USB-C ports only — a USB-C keyboard or a USB-A adapter must be physically on hand before the first boot, not sourced after a failure.

## Goals / Non-Goals

**Goals:**
- Add pyrite to the fleet as a NixOS machine with a full laptop hardware surface.
- Encrypt the root with a passphrase the operator types on the internal keyboard at boot.
- Produce a bare-metal install path that is re-runnable, not a one-off manual sequence.
- Reach travel-readiness: the machine is usable away from the ZeroTier mesh and away from a build host.
- Provide a local graphical desktop session. The machine boots to a GDM greeter on its internal panel, and an admin login reaches an interactive GNOME desktop with no network — which is what travel-readiness (Goal 4) means for a laptop, whose primary interface is its own screen rather than an ssh session from another host. niri is the eventual daily-driver target and is deferred to a reversible follow-up (D19); this goal is met by the stock GNOME desktop proven on this exact hardware from the installer ISO.

**Non-Goals:**
- Audio. The only out-of-tree module carrying the CS8409 amplifier init tops out below this repository's pinned kernel, and pinning the kernel backward for audio is ruled out. Revisit if mainline lands that init: sound then arrives on a routine kernel bump at no cost, which is why the hardware inventory note carries it as an upstream watch-item.
- Suspend/resume and the `d3cold` workaround. Deferred to a separate change; necessity unverified on this unit.
- Hibernation. Deferred, which is what makes D6 costless today.
- Correcting `base`'s cloud-VM initrd assumptions for the fleet. pyrite overrides per-machine; gating the shared module is a separate change touching five machines.
- Regenerating facter reports on this hardware. This change consumes a static report and does not regenerate one; regeneration itself is no longer blocked, now that `modules/nixpkgs/overlays/nixos-facter.nix` ships the nixos-facter#672 fix (see Open risks).
- A terranix entry for pyrite. Nothing reads one.
- Fixing the pre-existing justfile defects a new machine walks into (`check-uncached-machine` hardcodes four hosts and already omits magnetite).
- The niri Wayland session and its shell assembly. GNOME under GDM is the desktop this change ships (D19); niri is the eventual daily-driver target and lands as its own reversible follow-up deployed via `clan machines update`. Everything niri's shell requires beyond the compositor — a status bar, launcher, notification daemon, screen locker, graphical polkit agent, wallpaper tool, clipboard manager, network applet, terminal, settings UI, `xwayland-satellite`, the function-key helpers, and a hand-authored `config.kdl` — is out of scope here and MUST NOT be assembled into this change.

## Decisions

### D1: ZFS native encryption, created from a keyfile and flipped to a boot-time prompt

- **Choice**: a ZFS `zroot` pool with a `root` dataset carrying `encryption = "aes-256-gcm"` and `keyformat = "passphrase"`, created with `keylocation = "file://<partitioning-secret path>"` and a `postCreateHook` running `zfs set keylocation="prompt" zroot/root`, so every subsequent boot prompts locally. The dataset layout matches the fleet's `root`, `root/nixos`, `root/home`, `root/nix`.
- **Rationale**: three independent supports. Fleet parity — all five NixOS machines root on ZFS with a `zroot` pool and none has a non-ZFS root; ext4-on-LUKS would make pyrite the only non-ZFS root, forfeiting snapshots, compression, and the dataset layout every sibling uses. Exact parity with the closest available precedent — Mic92 is a clan-core developer, and both of his encrypted laptops (`machines/turingmachine/modules/disko.nix:47-56` and `machines/jacquardmachine/disko.nix:44-51`) are ZFS native, `aes-256-gcm`, `keyformat = "passphrase"`, `keylocation = "prompt"`, `ashift = "12"`, unlocked by a typed passphrase with no initrd SSH (only his server `eve` has that, `machines/eve/modules/network.nix:46`). And attestation — the create-from-keyfile-then-flip idiom is disko's own documented example, not an invention of this change.
- **Alternatives considered**: ext4-on-LUKS. An earlier revision of this design recommended it, on the premise that ZFS native encryption structurally requires a key pushed at every boot by a second machine and therefore could not have a boot-time prompt under a non-interactive install. That premise is false — see the refuted verdict above — and with it removed LUKS's only advantage disappears while its filesystem-inconsistency cost remains.
- **Accepted cost, stated plainly**: ZFS native encryption permits exactly one key per encryption root. `zfs change-key` replaces the key; it cannot add a second one. There is therefore no recovery passphrase, no escrow key, and no future `systemd-cryptenroll` path to a TPM or a FIDO2 token — those are LUKS keyslot features and ZFS has no keyslots. Additionally, ZFS native encryption does not encrypt pool layout, dataset names, or snapshot names; an attacker with the disk learns the dataset structure. This trade was put explicitly against the LUKS alternative, which offers keyslots and metadata encryption — precisely what is forfeited here — and which the preceding bullet records as considered and rejected.
- **Consequence for hibernation**: none today, because hibernation is a non-goal. See D6.

### D2: `ashift = "12"`, set explicitly

- **Choice**: `zpool.zroot.options.ashift = "12"`.
- **Rationale**: the disk reports 4096-byte logical and physical sectors, and 2^12 = 4096. Disko performs no sector-size detection and passes zpool options through verbatim to `zpool create -o`, so this is entirely the caller's responsibility. It is also unanimous across all seven of Mic92's pools. ZFS's `ashift=0` autodetect should arrive at 12 as well, but that was not grounded against OpenZFS source and is not relied on.
- **Note**: no existing machine in this repository sets `ashift`, because all five are 512-byte-sector cloud disks. pyrite is the first, and the resulting inconsistency is visible but not a defect in the others.

### D3: namespace-explicit `by-id` device path, and `boot.zfs.devNodes = "/dev/disk/by-id"`

- **Choice**: `device = "/dev/disk/by-id/nvme-APPLE_SSD_AP0512J_C08843605KKHV4MAK_1"`.
- **Rationale**: the controller exposes two namespaces. `nvme0n1` is the 465.9 GiB disk; `nvme0n2` is 8 KiB and must never be written. Their by-id names differ only by a `_1`/`_2` suffix, and the unsuffixed name is a bare prefix of both. `_1` and the unsuffixed name were both observed resolving to `nvme0n1`; `_1` is chosen because it names the namespace rather than relying on which namespace the unsuffixed alias points at.
- **Alternatives considered**: the unsuffixed name (rejected: prefix-ambiguous with the dangerous namespace); the wwid form `nvme-nvme.106b-...-00000001` (namespace-explicit and equally correct, rejected as unreadable at a review site where readability is the safety property).
- **devNodes**: every existing machine sets `by-path` with a comment reading "more stable for cloud VMs". That reasoning does not transfer to a laptop with a stable by-id path, and by-id is the bare-metal norm. ZFS scanning the 8 KiB namespace during import is harmless; the hazard is a write, and only the disko `device` writes.

### D4: the create-time key is a clan vars generator producing a human-typeable passphrase

- **Choice**: `clan.core.vars.generators.zfs` with `files.key.neededFor = "partitioning"`, whose script emits a human-typeable passphrase in the spirit of clan-infra `build01`'s `xkcdpass --numwords 6 --random-delimiters --case random` (`machines/build01/disko.nix:71-80`). The dataset's create-time `keylocation` points at that generator's `files.key.path`.
- **Rationale**: this is the channel `clan machines install` is built around, verified at the clan-core revision pinned in `flake.lock` (`d332b6935fbebebc0ca151efe0b3144f8dcd9d96`). `nixosModules/clanCore/vars/secret/sops/default.nix:30-32` resolves the `path` of any file with `neededFor == "partitioning"` to `/run/partitioning-secrets/<generator>/<file>`, and `pkgs/clan-cli/clan_lib/machines/install.py:171-182` rglob-walks that generated file tree and appends one `--disk-encryption-keys /run/partitioning-secrets/<...> <local path>` pair per file to the `nixos-anywhere` invocation automatically. Nothing is passed by hand.
- **`keyformat` MUST be `passphrase`, not `hex`**: clan-infra's `web01` uses `keyformat = "hex"` with `dd if=/dev/urandom | xxd` because it is a server unlocked over the network by a machine. A hex key is not usable by a person standing at a boot prompt. `build01`'s `xkcdpass` generator is the right shape; only its consumer (LUKS `passwordFile`) differs.
- **Why the flip is necessary rather than cosmetic**: the `/run/partitioning-secrets/...` path that `.path` resolves to exists only during the install, because nixos-anywhere places it there. It does not exist at boot. Leaving `keylocation` at that `file://` value would produce a machine that cannot find its key — which is precisely why web01 carries an initrd unit spinning until the file appears from elsewhere. The `postCreateHook` flip to `prompt` is what makes the boot self-sufficient. One consequence of the flip is worth recording rather than leaving to be discovered: once it has run nothing on the machine reads this var, because the boot path takes the same material from the prompt, so what stays committed is an install-time artifact whose only remaining consumer is a person reading it back.
- **Re-run idempotency**: verified against disko `lib/types/zfs_fs.nix:94-114`. On a first run the `zfs get type` probe fails and the create branch runs `zfs create -up ... -o keylocation=file://...`; on a re-run against an existing dataset the probe succeeds and the else branch runs `zfs set -u <updateOptions>`, where `updateOptions` is the create options minus `onetimeProperties` and therefore still contains `keylocation=file://...`. In both cases the `postCreateHook` runs afterward — `lib/default.nix:507-511` splices it after the create body unconditionally — and sets `keylocation=prompt`. The net state is `prompt` on either path.

### D5: import the nixos-hardware profile, disable two unwanted pulls

- **Choice**: add `nixos-hardware` as a flake input with `inputs.nixpkgs.follows = "nixpkgs"` mirroring disko's declaration at `flake.nix:83-84`, import `nixos-hardware.nixosModules.apple-macbook-pro-14-1`, and set `networking.enableB43Firmware = false` and `hardware.facetimehd.enable = false`.
- **Plain `false`, not `mkForce`**: both upstream values are `lib.mkDefault` — `apple/macbook-pro/14-1/default.nix:50` sets `networking.enableB43Firmware = lib.mkDefault true` and `apple/default.nix:4` sets `hardware.facetimehd.enable = lib.mkDefault (config.nixpkgs.config.allowUnfree or false)`. A plain definition overrides `mkDefault` at normal priority. An earlier revision of this document claimed `mkForce` was required because the values arrive from inside an imported profile; that is not how the module system's priority resolution works, and `mkForce` here would be unnecessary noise that also suppresses any future legitimate override.
- **b43**: a misdetection, and the reason it is declined is the silicon, not the licence. `networking.enableB43Firmware` (`nixos/modules/hardware/network/b43.nix:17`) pulls `b43Firmware_5_1_138` (`:30`), which serves the older SoftMAC BCM43xx parts; this machine's WiFi is a BCM4350 driven by `brcmfmac`, which that firmware does not serve. This is a correction, not a workaround: the claim that the pull hard-fails evaluation is false, because `modules/nixpkgs/base-defaults.nix:22` sets `allowUnfree = true` fleet-wide and it evaluates. Note precisely what the licence does and does not say. `b43Firmware_5_1_138` is `lib.licenses.unfree` (`pkgs/os-specific/linux/firmware/b43-firmware/5.1.138.nix:26`), evaluating to `{ free = false; redistributable = false; }`, and it is the `redistributable = false` half that keeps Hydra from building it and cache.nixos.org from serving it; `allowUnfree` governs neither half. Declining a package for foreign silicon is not a general principle against firmware — D15 affirms the redistributable firmware this machine's only NIC actually needs, and the two decisions sort on redistributability rather than on freeness.
- **facetimehd**: the fleet's global `allowUnfree = true` resolves the profile's `mkDefault` to true, auto-enabling an out-of-tree kernel module and unfree firmware on import. Disabled; the camera is not wanted and was not exercised during recon.
- **Consequence**: non-redistributable firmware is absent from binary caches and therefore builds locally, producing a store path signed by no trusted key. That is consistent with the store-signature failure observed during verification when pushing to the remote linux builder, though the failure was not reproduced against this package and is not asserted as its cause. Redistributable firmware raises no such question: it is cached, so it is fetched rather than built. The property doing the work here is redistributability and not freeness — `linux-firmware`, which D15 affirms, is a proprietary blob that caches serve normally — so this consequence is not a reason to decline firmware generally.
- **No "skip the profile" fallback**: an earlier revision offered setting the four initrd modules directly and skipping the profile. The fallback stays withdrawn, on the SPI/SMC input path rather than on the framebuffer. A two-way evaluation carrying pyrite's facter report and differing only by the import gives `boot.initrd.kernelModules = ["dm_mod" "i915"]` without the profile and `["applesmc" "applespi" "dm_mod" "i915" "intel_lpss_pci" "spi_pxa2xx_platform"]` with it, so the profile's marginal contribution to the option is exactly `applespi`, `spi_pxa2xx_platform`, `intel_lpss_pci`, and `applesmc` (`apple/macbook-pro/14-1/default.nix:19-24`), and nothing else in the planned module set puts any of them in the initrd. The diff isolates the profile: neither side carries `base`'s virtio pair or facter's `brcmfmac`, which the machine's full module set adds and which bear on nothing here. facter is structurally blind to the four rather than incidentally silent — `nixos/modules/hardware/facter/keyboard.nix:10,19` collects from `usb_controller` alone and writes to `availableKernelModules`, and this keyboard is SPI, while `intel_lpss_pci` does appear in pyrite's report but only under `.hardware.pci`, which no facter module routes into initrd. Hand-copying the four is possible, and it means owning them: a per-machine literal transcribed from an upstream file, with nothing to notice when upstream changes it. `hardware.intelgpu` cannot be hand-copied at all — `common/gpu/intel/kaby-lake/default.nix:10-13` sets it and it resolves to three `extraPackages`, but nixpkgs declares no such option and `nixos-hardware` declares it at `common/gpu/intel/default.nix:8-54`, so copying its effect means re-deriving a ~110-line package-selection module to its concrete answer and maintaining that answer by hand. The profile also carries `intel_iommu=on` (`14-1/default.nix:25`) and the `i915.enable_guc=2`/`i915.enable_fbc=1`/`i915.enable_psr=2` tunables (`common/gpu/intel/kaby-lake/default.nix:4-8`) into `boot.kernelParams`, which is otherwise `["root=fstab" "loglevel=4" "lsm=landlock,yama,bpf"]`, and a 17-line libinput quirks file (`14-1/default.nix:31-47`) whose three stanzas cover the SPI touchpad, the SPI keyboard, and the Touch Bar. The failure mode of skipping the profile is therefore not an invisible prompt but a worse and more specific one: the prompt renders and cannot be typed into.
- **The ground the fallback used to be declined on, and the one `i915` argument that survives**: the withdrawal was previously argued from `i915` reaching initrd only through the profile's import chain, so that hand-copying the SPI modules while silently dropping `i915` would produce an eval-clean, build-clean configuration with an invisible prompt. That scenario cannot occur and the "only" is false. `nixos/modules/module-list.nix:70` imports `./hardware/facter` into every NixOS configuration unconditionally, `hardware/facter/default.nix:16` imports `./graphics`, and `graphics/default.nix:33` assigns `boot.initrd.kernelModules = config.hardware.facter.detected.boot.graphics.kernelModules` at plain priority — not `availableKernelModules`, and not `mkDefault`. Pyrite's report passes both gates: `facter.enable` defaults to `report != {}` (`facter/default.nix:29`), `detected.graphics.enable` defaults to `length (report.hardware.monitor or []) > 0` (`graphics/default.nix:12`) and the report's `hardware.monitor` has one entry, and its `hardware.graphics_card[0].driver_modules` is `["i915"]`, which survives the `lib.remove "nouveau"` at `graphics/default.nix:19`. clan-core supplies the report to that module: `nixosModules/clanCore/default.nix:20` imports `./nixos-facter.nix`, whose `:10` sets `hardware.facter.reportPath` when the path exists, and task 4.7 lands the report at `machines/pyrite/facter.json`. The decisive evidence needs no counterfactual at all — cinnabar imports no nixos-hardware profile and evaluates `hardware.facter.detected.boot.graphics.kernelModules = ["virtio_pci"]`, the value `graphics/default.nix:33` assigns, and `virtio_pci` is in its `boot.initrd.kernelModules`; the module is firing in production in this fleet today. No repo-level `mkForce` could rescue the claim either, because this repository's only definition of the option is additive (`modules/system/initrd-networking.nix:34`). What survives is narrower, and it is a contingency rather than an exclusivity: the profile's `i915` is unconditional, since `loadInInitrd` defaults true (`common/gpu/intel/default.nix:19-23`), while facter's is gated on the report carrying a monitor entry, so a report regenerated with the lid closed drops it. That is also why the grounds above rest on the SPI modules rather than on facter — the facter module validates no report schema, since `collectDrivers` (`nixos/modules/hardware/facter/lib.nix:31`) reads `value.driver_modules or []`, so an upstream field rename would contribute an empty list with no evaluation error. Facter refutes the "only" decisively today while being a poor thing to depend on deliberately tomorrow.

### D6: keep `forceImportRoot = true`

- **Choice**: keep the `true` that `base` supplies, and record it as a decision rather than an inheritance.
- **Rationale**: it is load-bearing for a re-runnable install, which is this change's binding acceptance criterion. Mic92's `nixosModules/zfs.nix:16-18` states the mechanism in a comment scoped to exactly this class of machine: "Single-machine laptops/desktops: allow import after rescue/installer touched the pool with a different hostid, avoiding an emergency shell on next boot." A re-run of the install path touches the pool from the installer environment; without `forceImportRoot` the next boot lands in an emergency shell. The remaining interlock is unclean-export, where `false` would block boot after every unclean shutdown until someone types `zfs_force=1` at the console — a poor trade on a laptop, where an unclean shutdown is a normal event rather than an incident.
- **Known cost, and why it is not a cost today**: nixpkgs asserts `unsafeAllowHibernation -> !forceImportRoot && !forceImportAll`. `true` therefore forecloses ZFS hibernation at evaluation time, and because `base` assigns the value plainly rather than with `mkDefault`, changing it later requires `lib.mkForce`. Hibernation is deferred, so this interlock blocks nothing in scope. If suspend-to-disk is later wanted, this decision is what must change.

### D7: `networking.hostId` is inherited, not pinned

- **Choice**: set nothing. pyrite inherits clan-core's `nixosModules/clanCore/zfs.nix:10` `networking.hostId = lib.mkDefault "8425e349"`, as the whole fleet already does.
- **Rationale**: clan-core's own comment states the reason and it applies exactly here — "Use the same default hostID as the NixOS install ISO and nixos-anywhere. This allows us to import zfs pool without using a force import." The hostid-mismatch condition that `forceImportRoot` exists to override is the one this default is chosen to prevent: the installer that creates the pool and the installed system that imports it present the same hostid. Pinning a machine-specific hostid would manufacture the mismatch this default avoids and would make D6 load-bearing for a reason it does not currently need to be.
- **Alternatives considered**: pin a pyrite-specific hostid, as disko's own `tests/zfs-encrypted-root.nix:8-9` does on both installer and system. Rejected: that test pins the same value on both sides precisely to keep them matched, which is what the clan-core default already achieves fleet-wide without a per-machine literal. Mic92's two laptops pin nothing and lean on `forceImportRoot`, which is the same posture this arrives at with the additional safety of a matched hostid.
- **Relationship to D6**: these are belt and braces, not redundancy. The matched hostid means force-import is not needed in the ordinary case; `forceImportRoot` covers the unclean-export case and any rescue environment that does not share the default.

### D8: a recorded, re-runnable install path that discards the disk at every offset

- **Choice**: the install path is `clan machines install` against a booted stock installer ISO with sshd, with `--update-hardware-config` left at its default of `none` and the pre-generated report committed instead. The path is recorded in the repository, not performed ad hoc. It is preceded by an explicit, recorded `blkdiscard` of the target disk. `blkdiscard` is on the installer's PATH: it ships in `util-linux.bin`, and nixpkgs' `nixos/modules/config/system-path.nix:38` lists `util-linux` among `corePackageNames`, which `:184-186` folds unconditionally into `environment.systemPackages` on every NixOS system, the installer ISO included. Phase 7 confirms this on the machine before relying on it.
- **Rationale**: the binding requirement is that the path run repeatedly. terranix's `null_resource` local-exec is the only recorded invocation in the repository and it is cloud-only; without a recorded analog, pyrite ships as a one-off manual install.
- **The default disko mode destroys, so the run is not create-only**: `clan_lib/machines/install.py` passes no `--disko-mode` — zero hits across clan-core at the pinned rev `d332b69` — so nixos-anywhere's default governs. `src/nixos-anywhere.sh:419` sets `diskoMode=disko` when the flag is unset and `:422` forms `diskoAttr="${diskoMode}Script"`, selecting disko's `diskoScript`, whose body at `lib/default.nix:889-894` is `_disko`, which `lib/default.nix:1094-1098` composes as `_legacyDestroy` then `_create` then `_mount`. The happy-path run therefore destroys before it creates. An earlier revision of this design asserted the opposite and built the wipe's justification on it; the mechanism it described was real but its premise was not, and both are corrected here.
- **The wipe is belt-and-braces on the happy path, retained because the happy path is not guaranteed**: `_legacyDestroy` runs without `set -e`, so a destroy that fails silently falls through to `_create`, which lands in exactly the end state the earlier revision feared — `lib/types/gpt.nix:282-284` runs `sgdisk --clear` only `if ! blkid "${config.device}"`, so an Apple GPT that survived the destroy causes the clear to be skipped and the subsequent `sgdisk --new` calls to re-typecode Apple's partitions in place, while `lib/types/filesystem.nix:54-58` skips `mkfs.vfat` on an ESP that already reports a `TYPE=`, and the machine boots Apple's 300 MiB ESP rather than the declared layout. The explicit wipe is also the only step whose success the operator can independently observe before committing to an irreversible install, and clan-core's own encrypted-root guide at `docs/src/guides/disk-encryption.md:84-88` prescribes a manual `blkdiscard` before `clan machines install`, so upstream treats a pre-wipe as normal practice for this scenario rather than as belt-and-braces.
- **The wipe is `blkdiscard`, not `sgdisk --zap-all`**: zapping the GPT before `clan machines install` is actively harmful on a re-run. disko's `disk-deactivate/disk-deactivate.jq:7-9` reaches `zpool destroy -f` and `zpool labelclear -f` only through a node whose `fstype` is `zfs_member` — a child partition — and `:71-78` enumerates children from `lsblk`. A disk with no partition table has no children, so the partition-level `wipefs` and `labelclear` never run; the whole-disk `wipefs` at `:38` and `dd bs=440` at `:40` touch disk offsets only, while ZFS labels live inside p2. disko then recreates its deterministic layout, p2 reappears at the same offset with its labels intact, `lib/types/zpool.nix:298`'s `zpool import -N -f "zroot"` succeeds, and `:299` reports "not creating zpool as a pool with that name already exists" — the run goes green while reusing the old pool under the old passphrase, which is the tautological green the re-runnability requirement forbids, produced by the step meant to prevent it. `blkdiscard` against the `_1` namespace path from D3 destroys labels at every offset and has no such hole. If `blkdiscard` is ever unavailable, the fallback order is absolute: `zpool labelclear -f <device>-part2`, then `sgdisk --zap-all`, then `wipefs -a`.

### D9: ZeroTier peer, with the cinnabar redeploy as an explicit step

- **Choice**: tags include `nixos` and `peer`; `deploy.targetHost = "root@pyrite.zt"`; no `allowedIps` entry.
- **Rationale**: ZeroTier admission is declarative and evaluated at build time. The controller folds over every inventory machine in the moon, controller, and peer roles and reads each one's `zerotier-ip-<name>-<instance>` public var. The `allowedIps` list holds only darwin and external members not managed by the clan service; a NixOS peer needs no entry there.
- **The step that is easy to miss**: cinnabar must be redeployed after pyrite's ZeroTier IP var exists, or the new peer is never admitted. Identity and IP are generated offline on the admin box without touching the target, so pyrite's address is knowable before the machine is ever installed.

### D10: the tor onion service is taken, and `tor.nix`'s `nixos` selector is left alone

- **Choice**: set nothing. `modules/clan/inventory/services/tor.nix` keeps `roles.server.tags."nixos"`, and pyrite's `nixos` tag selects the tor server role exactly as it does on the fleet's five cloud hosts. A future NixOS machine takes the service by the same route, which this decision makes the intended default rather than a property of the selector nobody chose.
- **Rationale**: an onion service makes a machine with no stable address reachable from anywhere it has internet, which is worth more on a laptop than on the cloud hosts that already have a static IP and a DNS record. pyrite sits behind NAT wherever it travels, and D9's ZeroTier arrangement is otherwise its only way in — nor is that arrangement self-contained, since the fleet declares only `controller` and `peer` roles (`modules/clan/inventory/services/zerotier.nix:11,24`) and no moon, so pyrite reaches the mesh through ZeroTier's public root servers over the same internet a Tor circuit would use. A v3 onion address is network-independent by construction: it survives reboots, reinstalls, and every network the machine attaches to.
- **It is not a relay, and the refutation is firmer than the citation the verdict above carried**: `nixos/modules/services/security/tor.nix:1298-1307` is `lib.mkIf (!cfg.relay.enable)` and force-disables relaying outright — `DirPort = lib.mkForce [ ]`, `ORPort = lib.mkForce [ ]`, `PublishServerDescriptor = lib.mkForce false` — under a comment stating the point is to avoid enabling Tor as a relay by accident. The verdict cited the `mkEnableOption` at `:531-545` and the gate it opens at `:1277-1297`; the force-disable block on the other side of that gate is the half it missed, and it is the decisive one, because it holds even against an `ORPort` left in `services.tor.settings` by hand. Evaluating a machine that already takes the role settles it without a counterfactual: cinnabar returns `relay.enable = false`, `ORPort = []`, `DirPort = []`, `PublishServerDescriptor = false`, and `ExitPolicy = ["reject *:*"]`. The machine never advertises itself to the directory authorities and carries no third-party traffic.
- **What it does instead, stated because the `relay` namespace misleads**: an always-on tor daemon — `systemd.services.tor` is `wantedBy = [ "multi-user.target" ]` with `Restart = "on-failure"`, running as `User = "tor"` and `Group = "tor"` (`tor.nix:1361-1373,1426`) — publishing one v3 onion service exposing port 22 and nothing else. `clanServices/tor/default.nix:63-70` defaults `portMapping` to `[ { port = 22; target.port = 22; } ]`, and cinnabar evaluates `relay.onionServices.clan_tor = { version = 3; map = [ { port = 22; target.port = 22; } ]; }`. The address derives from an ed25519 key the generator mints offline on the admin box — `clanServices/tor/default.nix:138-144` runs `tor` for two seconds against a throwaway `torrc` and moves `hs_ed25519_secret_key` into `$out` — and commits it sops-encrypted; `vars/per-machine/<machine>/tor_tor/{hostname,hs_ed25519_secret_key}/secret` is tracked for six machines in this repository already. The hostname is itself a secret by default (`secretHostname`, `:78-91`), so the posture is address-confidentiality in front of a key-only sshd, which cinnabar evaluates as `PasswordAuthentication = false` and `PermitRootLogin = "prohibit-password"`. There is no client authorization: the address is withheld, not gated.
- **The precedent, and it is a sample of one**: Mic92 is a clan-core maintainer and he runs this on all four of his NixOS laptops through the identical selector. `machines/inventory.nix:129` is `tor.roles.server.tags.nixos = { };`, character for character what this repository has, and nothing filters it — evaluating his flake gives `.#nixosConfigurations.turingmachine.config.services.tor` as `{ enable = true; onions = [ "clan_tor" "ssh" ]; relay = false; }`, and turingmachine is a Framework laptop (`machines/turingmachine/configuration.nix:11`), with bernie, jacquardmachine, and dorits-laptop the other three. He does more than tolerate it: `nixosModules/sshd/tor.nix:6` adds a second ssh onion service of his own, and `:17-18` set `HiddenServiceNonAnonymousMode` and `HiddenServiceSingleHopMode`, deliberately trading the service's anonymity for speed. He is using onion services as a NAT-traversal reachability tool, which is the ground this decision takes them on.
- **The qualifier belongs with the evidence rather than in a footnote**: n is one. clan-infra offers nothing to compare against — `rg -i '\btor\b|onion' -g '*.nix'` over its whole tree exits 1 — which is a clean null result rather than a rejection, since every machine it manages is a server on a public IP with no NAT to traverse. The four other personal-clan references this repository's own CLAUDE.md names are not on disk, so no wider sample was obtainable. And the Mic92 clone is shallow at depth 1 (HEAD `7e55c8a4`), so whether he ever excluded laptops and reverted is not knowable from it. One maintainer running it deliberately on four laptops is the best available evidence and it is still one data point.
- **Accepted cost, stated plainly**: the machine announces a stable, permanent, network-independent identity to the Tor network from every network it attaches to — café, hotel, conference — using unobfuscated Tor with no bridge and no pluggable transport, maintaining outbound circuits and periodically republishing its descriptor. The address is withheld rather than gated, so its confidentiality is the whole of the access control in front of sshd. That is the cost, and it is accepted.
- **An earlier revision declined the role, on a preference that was never stated**: that revision's Alternatives-considered bullet rejected keeping the tag because "the user does not want an ssh endpoint of a travelling laptop published to Tor". The record does not carry that preference. Filtering both session transcripts to human utterances yields 28 messages, and searching them for tor, onion, hidden service, publish, expose, or travel returns one hit — the user pasting a prior agent's prose about `nixos-hardware.nixosModules.apple-macbook-pro-14-1` being "exposed as" a flake output, which is neither about Tor nor his own words. The question was in fact put to him twice, and mischaracterised both times as enrolling the machine as a Tor relay: "tagging `nixos` auto-enrolls the machine as a Tor relay. On your travel laptop.", and "…as a Tor relay. On a laptop you travel with. Almost certainly not what you want, but it's your fleet's tag semantics and I won't silently opt you out." That relay claim is false, as the verdict above records, so the silence that followed was silence in response to a question about a service this role does not produce. When the relay claim collapsed the role was declined anyway, and the basis was described accurately at the time — "the role is declined for pyrite anyway — an onion-ssh service on a travel laptop wasn't asked for". "Wasn't asked for" is a statement about the record; "the user does not want" is a claim about his mind; the revision converted the first into the second. The attribution is withdrawn, and the role is decided on the merits above.
- **Alternatives considered**: narrow the selector to an explicit `roles.server.machines` list of the five cloud hosts, which is what the withdrawn revision chose (rejected — its grounds do not survive, and the narrowing carries its own cost, moving the five cloud hosts from tag-selected to name-selected and adding a hand-maintained list of exactly the kind `flake-shape.nix` already imposes, which a future machine can be forgotten from). Drop the `nixos` tag from pyrite (rejected — `modules/clan/inventory/services/sshd.nix` selects both its server and client roles by the same tag, so dropping it silently forfeits persistent host keys and CA-signed host certificates; clan-core's inventory offers no per-machine exclusion from a tag-selected role, which is why the selector was the only other lever).

### D13: the install mechanics, as they actually resolve

- **nixos-anywhere is 1.13.0, resolved through `clan-core.inputs.nixpkgs`**: it is neither vendored, nor run via `nix run`, nor taken from `PATH`. `flake.lock` resolves `clan-core` to `d332b69` and its `nixpkgs` input to the `nixpkgs_2` node at rev `173d0ad7a974`, whose `nixos-anywhere` is version `1.13.0` (upstream rev `bad98b0`). Every behaviour recorded in D8 and here was read from that version's `src/nixos-anywhere.sh`, not from upstream `main`, and the two differ.
- **Do not run `nix flake update` between now and the install**: `flake.nix:70` floats `clan-core` on `https://git.clan.lol/clan/clan-core/archive/main.tar.gz`. Any update can move `clan-core`, and with it the transitive `nixpkgs_2` node and nixos-anywhere's version, without a `nixos-anywhere` entry appearing in the diff — there is no such node in `flake.lock` to inspect. The mechanics recorded here hold for 1.13.0 and are not carried by the lock in a form a reviewer can see.
- **clan invokes nixos-anywhere four times, once per phase**: `clan_lib/machines/install.py:227-254` defines `run_phase`, which appends `--phases <phase>` to the assembled argv, and calls it in turn for `kexec`, `disko`, `install`, and `reboot`. The earlier `--phases` append at `:187-188` is inert for argv resolution — the per-phase append comes last and bash last-wins resolves it — but calling it dead code overstates the case, because `opts.phases` is load-bearing for which phases `run_phase` is called for at all. What the last-wins resolution makes inert is the earlier flag's effect on the assembled command line, not the option's effect on the set of phases that run.
- **Keep the default phases**: `kexec` is a no-op against an installer ISO, since `src/nixos-anywhere.sh:694-697` returns immediately from `runKexec` when `isInstaller == "y"`. Dropping it is not free: `:978-983` rewrites `sshConnection` to `root@${sshHost}` whenever the `kexec` phase is absent, and does so before `uploadSshKey` at `:983`, which breaks sudo escalation from a non-root installer session.
- **Omitting `--update-hardware-config` is safe**: clan's `--update-hardware-config` defaults to `none` (`clan_cli/machines/install.py:280`) and clan appends `--generate-hardware-config` only when it is not `none`. nixos-anywhere's own `hardwareConfigBackend` also defaults to `none` (`src/nixos-anywhere.sh:32`). The two interactive prompts that would otherwise offer to generate a report both short-circuit on `args.yes`, so `--yes` bypasses them. facter never runs on the target and the pre-committed report is what the build consumes.
- **Do not pass `--force-kexec`**: the flag does not exist at 1.13.0, so passing it aborts. It exists on upstream `main`, where `src/nixos-anywhere.sh:280-283` pairs `forceKexec=true` with a `shift` that a boolean flag does not need, consuming the following argument. It is unusable in both directions.

### D14: network association through clan-core's wifi clanService, with the credentials as clan vars

- **Choice**: instance clan-core's first-party wifi service — `clanServices/wifi/default.nix`, exported as `clan.modules.wifi` at `clanServices/wifi/flake-module.nix:8` — from a new `modules/clan/inventory/services/wifi.nix` targeting `roles.default.machines."pyrite"` with a single network. The SSID and the PSK become clan vars, sops-encrypted and committed, prompted by the same `clan vars generate pyrite` that D4's disk passphrase already routes through.
- **Rationale**: the credentials become repository state rather than machine state, which is the property the install path requires and the reason both alternatives below fail. The service is first-party and the instance takes the shape every other file in `modules/clan/inventory/services/` already uses, so nothing bespoke is introduced. Targeting is machine-scoped rather than tag-scoped because no `wireless` tag exists and pyrite is the sole taker.
- **How the association actually works**: the generators at `clanServices/wifi/default.nix:76-90` declare `network-name` with `type = "line"` and `password` with `type = "hidden"`, both `persist = true`, and both prompt interactively at generate time. The SSID is a var rather than a literal because the NetworkManager profile is a world-readable store path and upstream cannot inline it. At activation `ensureProfiles.environmentFiles` points at `/run/secrets/NetworkManager/wifi-secrets` (`:98-100`), one `ensureProfiles.profiles.<name>` is emitted per network (`:102-115`), and a `NetworkManager-setup-secrets-wifi` oneshot writes the envsubst variables ordered before `NetworkManager-ensure-profiles` (`:119-146`). `autoConnect` defaults true (`:30-33`) and lands as `connection.autoconnect` (`:107`), so a bare `networks.fleet = { }` associates unattended at first boot — provided the vars were generated and committed before the deploy. If they were not, the profile interpolates empty strings and association fails silently; no assertion and no eval-time error guards it. Task 7.7 carries this.
- **`share = true` is not overridable, and on this instance it is also what is wanted**: the per-network generator carries `share = true` (`:88`) — one generator holding both prompts, not one generator per credential — so the credentials land under `vars/shared/wifi.<name>/` rather than `vars/per-machine/pyrite/`, per the `shared` path fragment `clan_lib/vars/_types.py:41-49` returns, and `roles.default.interface` exposes only `enable`, `autoConnect`, and `keyMgmt`, so no instance setting declines it. Nothing here wants to decline it. The fleet SSID exists to serve the fleet, so a second machine joining that network should take the same credential; sharing is the intent rather than a cost absorbed in order to use the service. Nor does sharing widen the recipient set, which is the alarm the word invites: `clan_lib/vars/generate.py:356-370` re-encrypts a shared secret only to the machines in `generator.machines`, so with pyrite the sole taker the recipients are exactly a per-machine var's, and they grow only as machines take the network — which is the same condition under which sharing is wanted. The residual is narrower than "shared" suggests and is worth naming exactly: a network's attribute name is a clan-wide identity naming one credential, so per-machine variation of the same named network is inexpressible, and two machines declaring that name with different settings raise `Machines {a} and {b} have different definitions for shared generator 'wifi.<name>'` at `clan_lib/vars/generator.py:294`. A second wireless host on a different network remains expressible as a differently-named network, which is the shape upstream's own VM test uses to run three networks across two machines (`clanServices/wifi/tests/vm/default.nix:12-37`). Shared vars are already an accepted shape here — `modules/clan/inventory/services/users/cameron.nix:29` carries `share = true` for the admin password across every machine taking that role.
- **What is committed is a dedicated fleet SSID's credential, and the test that admits it is origination**: the network pyrite associates with is one this repository stands up for the fleet rather than the household network, and the SSID and PSK committed here are that network's. The organizing principle is that this repository commits secrets it originates and withholds secrets it borrows. Neither reference repository states the test and both follow it. clan-infra commits its disk secrets because it makes them — `machines/web01/disko.nix:49-58` generates a key with `dd if=/dev/urandom bs=32 count=1 | xxd -c32 -p` and `machines/build01/disko.nix:71-80` generates a passphrase with `xkcdpass --numwords 6` — and the resulting ciphertext at `vars/per-machine/{web01,build01}/zfs/key/secret` is load-bearing rather than archival, since `machines/web01/decrypt.sh` pushes that key over initrd SSH at every reboot. Mic92 commits exactly one wifi PSK: his OpenWRT access point's own key, declared at `openwrt/example.nix:91-108` with `mode = "ap"` and `key = "@wifi_password@"` across the SSIDs `gchq-2.4` and `gchq-5`, and held as sops ciphertext at `openwrt/secrets.yml:1`. He commits that one because there the repository defines the network. A fleet SSID this repository stands up is originated in the same sense: the operator chooses the value and configures the router to match it, so the dependency runs from this repository to the network rather than the reverse, and rotating it is a unilateral act with no other party's devices to re-pair. Origination is authorship of the value and not the mechanism that produces it — Mic92's OpenWRT key is a hand-placed sops value with no generator behind it, and this service prompts rather than generates, which is why task 5.1 has the operator make the PSK out of band and task 5.2 types it in. An earlier revision of this bullet grounded the commit on D4's precedent instead, reasoning that this change already commits pyrite's ZFS passphrase through this mechanism and that a PSK is therefore the smaller of two exposures. That reasoning is withdrawn: D4's passphrase is a secret this repository originates, so it licenses committing other secrets this repository originates and says nothing about a secret this repository borrows. The axis is origination, not magnitude.
- **The household network's PSK is not committed, and publication rather than sensitivity is what fixes the line there**: under the origination test the household network is borrowed — someone else defined it, other people's devices depend on it, and this repository's only relation to it is that pyrite might want to join. `github.com/cameronraysmith/vanixiets` is public, which `gh repo view` reports as `"visibility": "PUBLIC"`, so any sops-age ciphertext committed here is world-readable and permanent, and because clones do not register as forks the blob outlives any later decision to remove it. A household credential is consequently not solely this operator's to publish, and rotating it after a mistake would mean re-pairing every device belonging to people who did not consent to the publication — an objection that does not turn on the strength of the ciphertext and does not dissolve if the ciphertext is strong. If pyrite ever needs the household network it associates interactively through `nmcli`, exactly as it would with any other network this repository does not define, and that credential's durable home is a password manager. That is where Mic92's laptops keep theirs.

### D15: `hardware.enableRedistributableFirmware = true` and `hardware.cpu.intel.updateMicrocode = true`, stated explicitly

- **Choice**: set both in the machine module, at `true`, rather than inheriting either.
- **Why stated rather than inherited**: `nixos/modules/hardware/facter/firmware.nix` gates its entire block on bare-metal detection and sets both as `mkDefault`. Every existing machine reports `kvm` or `google` and pyrite reports `virtualisation = "none"`, so that branch has never evaluated in this repository and comes live here for the first time — the facter-placement verdict above records it as one of three newly-live code paths. A value arriving from a branch no one in this fleet has evaluated is an inherited default, not a decision. Stating both makes the machine's firmware posture legible where a reader looks for it, and independent of whether facter's detection fires.
- **The axis is redistributability, not free versus unfree**: this is the distinction the rest of the decision turns on, and having it backwards is what makes D5 read as contradicting this one. `hardware.enableRedistributableFirmware` (`nixos/modules/hardware/all-firmware.nix:52-53`) is described as covering "firmware with a license allowing redistribution", and its config block at `:71-86` sets `hardware.firmware = [ linux-firmware ... ]` with linux-firmware at `:75`. `linux-firmware` carries `unfreeRedistributableFirmware` (`pkgs/by-name/li/linux-firmware/package.nix:58`), which evaluates to `{ free = true; redistributable = true; }` — nixpkgs deliberately treats redistributable firmware as free for channel inclusion. `b43Firmware_5_1_138`, which D5 declines, carries `lib.licenses.unfree` (`pkgs/os-specific/linux/firmware/b43-firmware/5.1.138.nix:26`), evaluating to `{ free = false; redistributable = false; }`. Both are proprietary binary blobs. What separates them is that one may be redistributed, so Hydra builds it and cache.nixos.org serves it, and the other may not, so it builds locally and lands as a store path signed by no trusted key. `allowUnfree` governs neither and is irrelevant to both; this repository sets it true at `modules/nixpkgs/base-defaults.nix:22` regardless.
- **The path this closes**: D5 declines two firmware pulls to keep blobs out of the closure, and worked twice that reads as a general principle. Applied to `hardware.enableRedistributableFirmware`, the principle produces `false`, and `false` is this machine's only NIC going dark. The option does not defend itself: its default is `config.hardware.enableAllFirmware` (`all-firmware.nix:55`), which is false absent another definition, so on a machine where facter's newly-live branch is the only thing setting it, an implementer minimizing the closure has an unobstructed path to a laptop with no network and no obvious cause. Stating `true` in the module, beside D5's two `false`s, is what forecloses it. The declines and this affirmation are not in tension because they do not sort on the same property: b43 firmware serves silicon this machine does not have and facetimehd drives a camera that is out of scope, while `linux-firmware` carries the blob the machine's only NIC needs in order to probe at all.
- **What the NIC needs, and that linux-firmware carries it**: the WiFi is `14e4:43a3`, driven by `brcmfmac` per the facter report, and it needs `brcm/brcmfmac4350-pcie.bin`, `brcm/brcmfmac4350c2-pcie.bin`, or both. Both are present in `linux-firmware` and listed in its `WHENCE` under `Driver: brcmfmac` with "Licence: Redistributable".
- **The decisive evidence is empirical rather than textual**: CAM-31's recon booted a stock NixOS installer ISO. That ISO imports `all-hardware.nix`, which sets `hardware.enableRedistributableFirmware = true` plainly at `nixos/modules/hardware/all-hardware.nix:171` and sets no other firmware option. Under exactly that configuration `brcmfmac` probed and `wlp2s0` appears in `pyrite-facter.json`. This unit has already booted with working wifi under this setting alone, which is a stronger warrant than any argument from licence metadata and is why the decision is not held open pending the install.
- **Residual, stated honestly**: `linux-firmware` ships no Apple-specific NVRAM blob for the 4350, so the NVRAM comes from the chip's OTP. The recon is evidence that it does so on this unit. That is the one part of the wifi firmware story resting on observation with no source behind it.
- **Microcode**: `hardware.cpu.intel.updateMicrocode` (`nixos/modules/hardware/cpu/intel-microcode.nix:14-20`) defaults false, and `microcode-intel` carries `unfreeRedistributableFirmware`, so it is fetched from a cache rather than built locally and raises none of the store-signature question D5's consequence records. On a Kaby Lake part there is no reason to decline it, and the same state-rather-than-inherit argument applies, since facter's bare-metal branch would otherwise supply it as a `mkDefault` nobody chose.
- **Alternatives considered**: leave both to facter's `mkDefault`s (rejected — it makes the machine's firmware posture depend on a detection branch this fleet has never exercised, and leaves nothing at the machine module to stop the closure-minimizing `false`); set `hardware.enableAllFirmware = true` instead (rejected — `enableRedistributableFirmware`'s default is `config.hardware.enableAllFirmware` at `all-firmware.nix:55`, so the broader option would satisfy this decision as a side effect, but its purpose is to reach beyond the redistributable set, which is firmware this machine has no use for, and it states the posture less precisely rather than more).

### D20: `system.stateVersion = "26.05"`, matching the release this machine is installed from

- **Choice**: `system.stateVersion = "26.05"`, written at plain priority in `modules/machines/nixos/pyrite/default.nix:56`, matching the 26.05 installer ISO release D18 settles. The value is not carried over from the electrum host module this one was derived from, which reads `25.05`. The committed `vars/per-machine/pyrite/state-version/version/value` is corrected to `26.05` to agree with it.
- **Rationale**: `system.stateVersion` records the release whose stateful defaults a machine was first brought up under, and it is never changed after install — nixpkgs' own option description says so in as many words (`nixos/modules/misc/version.nix` at the pinned nixpkgs): "This option defines the first version of NixOS you have installed on this particular machine", and "Most users should **never** change this value after the initial install, for any reason, even if you've upgraded your system to a new NixOS release." That is why it is settled here, before the physical install, rather than corrected afterward: every other value in this document can be revised by a `clan machines update` over ZeroTier, and this one cannot. A machine installed from the 26.05 ISO and carrying `25.05` would claim compatibility with defaults it was never brought up under, and the claim would be unfalsifiable from the running system.
- **The value is per-machine, not a fleet-wide constant**: cinnabar, electrum, galena, magnetite, and scheelite retain `25.05`, and that is correct for them, because it is the release each of those machines was installed from. Their vars are left untouched by this change. Divergence between machines is the option behaving as specified, and a repository-wide sweep to one value would be the defect.
- **The clan interaction, and why the var is corrected rather than left inert**: clan-core's state-version module sets `system.stateVersion = lib.mkDefault (lib.removeSuffix "\n" var.value)` from the committed var (`nixosModules/clanCore/state-version/default.nix:18`), so pyrite's plain-priority assignment outranks it and `nix eval --raw .#nixosConfigurations.pyrite.config.system.stateVersion` returns `26.05` whatever the var holds. The var was generated while the host module still read `25.05`, because the generator records the resolved value at generation time — `:26` is `echo -n ${config.system.stateVersion} > "$out"/version` — and it retained that value after the module changed. Correcting it is not required for evaluation and is done anyway, on two grounds: the generator declares no `validation`, so clan never reports the divergence, and removing the explicit assignment as apparently-redundant would then silently revert an installed machine to `25.05`, which is the one edit this option cannot survive.

## Invariants

### pyrite MUST NOT `mkForce` `boot.initrd.kernelModules`

`base` contributes `virtio_pci` and `virtio_net` to `boot.initrd.kernelModules` at `modules/system/initrd-networking.nix:33-37`, and those entries are meaningless on bare metal.
The natural-looking way to drop them is `boot.initrd.kernelModules = lib.mkForce [ ... ]`, and that is a lockout.
The option is a list that accumulates definitions from more sources than this document can close over, and the ones named here are examples rather than a set.
Among the contributors this design imports deliberately: `base`'s virtio pair (`modules/system/initrd-networking.nix:33-37`), the nixos-hardware profile's `applespi`/`spi_pxa2xx_platform`/`intel_lpss_pci`/`applesmc` (`apple/macbook-pro/14-1/default.nix:19-24`), `common/gpu/intel`'s `i915` (`common/gpu/intel/default.nix:90`), facter's `brcmfmac` (`hardware/facter/networking/initrd.nix`), and facter's own `i915` (`hardware/facter/graphics/default.nix:33`), which is a definition distinct from `common/gpu/intel`'s and arrives at plain priority whether or not the profile is imported.
Among the contributors nobody imported, because `nixos/modules/module-list.nix` loads them into every NixOS configuration: `dm_mod` (`system/boot/kernel.nix:379`, module-list `:1927`), gated on `boot.initrd.includeDefaultModules`, whose default is `true` at `kernel.nix:253-255`; `af_packet` (`system/boot/initrd-network.nix:124`, module-list `:1924`), gated on the `boot.initrd.network.enable` that `base` sets; and `zfs` (`tasks/filesystems/zfs.nix:726`, module-list `:2010`), gated on the root being ZFS, which pyrite's is.
The two other nixpkgs modules mentioning `dm_mod` contribute nothing to it: `system/boot/luksroot.nix:1142` and `system/boot/systemd/dm-verity.nix:32` both write `availableKernelModules`, the weaker option this document distinguishes from the force-loading one throughout.
Cinnabar settles the shape and needs no counterfactual: `nix eval --json .#nixosConfigurations.cinnabar.config.boot.initrd.kernelModules` returns `["af_packet","dm_mod","virtio_balloon","virtio_console","virtio_gpu","virtio_net","virtio_pci","virtio_rng","zfs"]`, and cinnabar imports no nixos-hardware profile — `nixos-hardware` is absent from `flake.lock` until task 1.1 lands it — so `dm_mod`, `af_packet`, and `zfs` reach that list on a machine whose only deliberate contribution to it is `base`'s virtio pair.
The set is not exhaustively enumerable here, and that is the argument rather than a gap in it: this document's count has been wrong twice, and the revision that produced the second count did not reach the three stock modules above.
`mkForce` discards every definition it does not name, and the definitions it would not name cannot be listed in advance — which is why the prohibition does not rest on the enumeration being complete, and is strengthened rather than weakened by its not being.
A list containing only the intended survivors would silently drop the SPI input modules, both `i915` definitions, `dm_mod`, or any combination, producing a configuration that evaluates cleanly, builds cleanly, and boots to a passphrase prompt that is either invisible or unanswerable on a machine with no macOS to fall back to.
The failure envelope is strictly larger than the one D5's withdrawn fallback carries, which is what earns this an invariant of its own rather than a restatement of D5: skipping the profile drops that import's definitions and leaves facter's, so the prompt still renders and merely cannot be typed into, whereas `mkForce` outranks every definition at once and can take the framebuffer away with the keyboard.

The virtio entries are inert on bare metal: they modprobe, find no matching device, and cost a few kilobytes of initrd.
They are left alone.
What pyrite overrides is `boot.initrd.network.ssh`, which is a distinct option and can be disabled without touching the module list.

### Whether disko's destroy phase reports failure reliably on this disk is unestablished

A candidate reason to distrust the destroy phase's exit status on this specific device was considered and could not be established.
The conjecture is that `disk-deactivate.jq:40`'s `dd if=/dev/zero of=<disk> bs=440 count=1` cannot write a partial sector on a 4096-byte-logical-sector disk, and so fails or misreports.
It is recorded here as unverified rather than as a rationale, because the reasoning runs the other way: `dd` opens the block device buffered, not `O_DIRECT`, and Linux services a sub-sector buffered write to a block device by read-modify-write of the containing block, which is why the `bs=440` MBR-zeroing idiom is used ubiquitously against 4Kn disks.
No 4Kn device was available to test against, and the facter report records no logical sector size for `nvme0n1`, so neither direction was confirmed by observation.
Nothing depends on the answer: the `blkdiscard` step in D8 stands on `_legacyDestroy`'s missing `set -e` and on operator-observability, both of which hold regardless, and Phase 7 records `blockdev --getss /dev/nvme0n1` on the installer, which settles the sector size by observation at a cost of one command.
If that check returns 4096 and a destroy-phase failure is then actually observed, the conjecture becomes worth filing upstream against disko; until then it is not.

### `canTouchEfiVariables = true` writes Apple firmware NVRAM and is untested against it

`boot.loader.efi.canTouchEfiVariables = true` (task 2.3) is the only step in this change that writes firmware NVRAM, and on this machine that NVRAM is an 8 MiB SPI ROM on the logic board (facter `smbios` `rom_size` 8388608), not on the disk the install wipes.
Every fleet machine carrying the setting — electrum (`:48`), scheelite (`:50`), galena (`:46`) — is a cloud VM on virtual OVMF firmware, so it is exercised nowhere against Apple firmware, and it appears in no decision and, until now, no open risk.
It is not a brick vector: the boot ROM, the Option-key picker, and firmware recovery are independent of the disk, so the worst case is an EFI write failing, recoverable by an NVRAM reset (Cmd-Opt-P-R, itself a boot-ROM feature).
If writes do fail on Apple firmware, the fallback is `canTouchEfiVariables = false`, which stops the install touching NVRAM at the cost of leaving boot-entry management to the firmware's own ESP fallback path.
The recovery floor is unchanged either way: the LaCie live ISO remains re-bootable from the Option-key picker on the internal keyboard (D18).
