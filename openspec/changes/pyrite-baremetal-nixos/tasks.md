## 6. Phase 6 — The recorded install path

- [x] 6.1 Write the install path as a recorded, re-runnable artifact (justfile recipe and/or a documented runbook), whose first step discards the disk with `blkdiscard` against the `_1` by-id path — following clan-core's own form at `docs/src/guides/disk-encryption.md:84-88` — and which then invokes `clan machines install pyrite --target-host nixos@<installer-ip> -i <key> --yes` with `--update-hardware-config` left at its default of `none`. The target is `nixos@`, not `root@`: the stock installer autologins the `nixos` user (`installation-device.nix:61`), places it in `wheel` (`:40`), and gives it passwordless sudo (`:57`), while both `nixos` and `root` ship empty passwords (`:45`, `:49`) that sshd refuses for password auth — so `root@` would need a key hand-installed into `/root/.ssh` first, whereas `nixos@` needs only the key authorized in the GNOME session already open on the machine, and nixos-anywhere escalates to root through that passwordless sudo. The installed-system `deploy.targetHost = "root@pyrite.zt"` (task 4.2) is unrelated and stays: it deploys over ZeroTier to the installed machine's own host key. Use `blkdiscard`, not `sgdisk --zap-all`. Under D1's container the reason is restated rather than carried over: p2's `fstype` is now `crypto_LUKS`, not `zfs_member`, so disko's `disk-deactivate.jq` cannot reach the `zpool destroy`/`labelclear` branch at `:7-9` for p2 by type at all, and what runs instead is a bare `wipefs --all -f` at `:42-45` that erases the primary LUKS2 signature while leaving the secondary header and the whole 16 MiB keyslot area intact — so the old passphrase and the old FIDO2 enrollment survive a wipe that reads as complete. Zapping the table first is worse still: with no children for `lsblk` to report the p2 arm never runs at all, and the next install finds a valid header, skips `luksFormat` (`lib/types/luks.nix:202`) and skips the FIDO2 enrollment (`:276`). If `blkdiscard` proves unavailable, the fallback order is absolute and it is no longer the ZFS one — `cryptsetup luksErase --batch-mode <device>-part2` first, because it needs a header that still probes; then `dd if=/dev/zero of=<device>-part2 bs=1M count=32`, covering both headers and the keyslot area rather than the primary signature alone; then `sgdisk --zap-all`; then `wipefs -a`. Drop `zpool labelclear` from that order rather than reordering it: under a container p2 holds no ZFS labels to clear, so it would succeed at nothing and read as progress
